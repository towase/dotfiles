## 開発スタイル

TDD で開発する（探索 → Red → Green → Refactoring）。
KPI やカバレッジ目標が与えられたら、達成するまで試行する。
不明瞭な指示は質問して明確にする。

## コード設計

- 関心の分離を保つ
- 状態とロジックを分離する
- 可読性と保守性を重視する
- コントラクト層（API/型）を厳密に定義し、実装層は再生成可能に保つ
- 静的検査可能なルールはプロンプトではなく、その環境の linter で記述する
- コードコメントには基本的に why や why not のみ記述すること。複雑なロジックの場合は何をやっているかを書いても良い。

## ユーザビリティ設計
- エラーメッセージはなるべく次のユーザーのアクションを伝える

## GitHub Issue ルール
- 達成条件はアウトカム（得たい成果）で記載する。成果が一意なアウトプットに還元できる場合に限り、アウトプットベースでも可

## GitHub PR ルール
- PRを作成する場合は、PR Template が存在する確認し、あれば PR Template に沿って作成してください。

## Git ブランチルール
- 作業開始時に現在のブランチが main または master の場合、必ず新しいフィーチャーブランチを作成してから作業を始めること
- ブランチ名は作業内容を反映した名前にすること（例: feat/add-login, fix/header-style）
- main / master ブランチに直接コミットしないこと

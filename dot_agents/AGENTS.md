## 開発スタイル

原則として TDD で開発する（探索 → Red → Green → Refactoring）。
ただし、既存のテスト戦略がないリポジトリでは、変更のリスクと既存慣習に見合う最小限の検証に留める。
テスト基盤やテスト方針を新設するのは、ロジック・状態遷移・回帰リスク・複数箇所に影響する契約がある場合に限る。
フロントエンドの静的な HTML/CSS や見た目だけの変更には、原則として専用テストを追加しない。必要に応じてブラウザ確認、スクリーンショット、既存の lint/build で検証する。
既存のテストがある場合はその枠組みを優先し、新しいテスト種別やライブラリを安易に追加しない。
KPI やカバレッジ目標が与えられたら、達成するまで試行する。
不明瞭な指示は質問して明確にする。

## コミュニケーション

- 次のものは、コンソール Markdown ではなく `/tmp/<topic>.html` に HTML ファイルとして書き出してパスを伝える。ユーザーはブラウザで開いて確認・回答する:
  - **確認**: 選択肢の提示・複数案の比較・レビュー結果など視覚的な確認を求める場合（AskUserQuestion も使わない）。HTML には背景、現状、選択肢（コードブロック + Pros/Cons）、AI 推奨案を含める。
  - **レポート**: 調査結果・根本原因分析・設計レビュー結果など、一定の分量や構造を持つもの。HTML には目的、調査結果（事実）、引用（コード・ファイルパス・SHA など）、結論または推奨アクションを含める。
- HTML ファイルを書き出したら、続けて Bash ツールで `open <filepath>` を実行し、デフォルトブラウザで自動的に開く。ユーザーにコピペさせない。コンソールには書き出したパスのみ伝える。

## コード設計

- 関心の分離を保つ
- 状態とロジックを分離する
- 可読性と保守性を重視する
- コントラクト層（API/型）を厳密に定義し、実装層は再生成可能に保つ
- 静的検査可能なルールはプロンプトではなく、その環境の linter で記述する
- コードコメントには基本的に why や why not のみ記述すること。複雑なロジックの場合は何をやっているかを書いても良い。

## ユーザビリティ設計
- エラーメッセージはなるべく次のユーザーのアクションを伝える

## GitHub Issue ルール
- 達成条件はアウトカム（得たい成果）で記載する。成果が一意なアウトプットに還元できる場合に限り、アウトプットベースでも可

## GitHub PR ルール
- PRを作成する場合は、PR Template が存在する確認し、あれば PR Template に沿って作成してください。

## GitHub PR レビュールール
- レビューの総評コメント（approve body 含む）は短く 1〜2 文に留める。確認ポイントの列挙や長文評価はレポート（`/tmp/*.html` や `.md`）に逃がし、PR 上には書かない
- approve を出すときは「merge 判断はお任せします」のように作者判断を尊重するトーンで書く。「LGTM!」「マージしてよいと思います」のような断定・推奨は使わない
- 個別の指摘は総評（approve body）に書き込まず、inline コメントまたは PR 全体コメントとして別建てで投稿する
- 指摘がスコープ拡張を含む場合、「本 PR のスコープに含めるか、別 PR・別 issue に分けるかはお任せします」のように、レビュアーの押し付けにならない言い回しを使う
- ユーザーが PR 作者でないとき、inline / 通常コメント問わず、投稿前に本文をユーザーに見せて承認を取る

## Git ブランチルール
- 作業開始時に現在のブランチが main または master の場合、必ず新しいフィーチャーブランチを作成してから作業を始めること
- ブランチ名は作業内容を反映した名前にすること（例: feat/add-login, fix/header-style）
- main / master ブランチに直接コミットしないこと

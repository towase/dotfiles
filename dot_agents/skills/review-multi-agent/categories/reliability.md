# reliability 観点（優先順位: 4）

## このカテゴリで見るもの
**「本番に出した後、想定外の事象が起きても**(1)**サービスが回り続ける**(2)**何が起きたか追跡できる**
**(3)**安全に戻せる」**ための設計判断を見る。具体的には、外部 I/O の失敗扱い・冪等性・整合性・
グレースフルデグラデーション・障害分離・ヘルスチェック・観測性（ログ・メトリクス・トレース）・
アラート/ランブック/SLO 意識など。

`robust-safety` がコードレベルの不正状態回避（型・並行・例外設計）を扱うのに対し、本カテゴリは
**システム全体の運用挙動**を扱う。`_reviewer-stance.md` B 節（段階的開発戦略）が「**安全に出荷できるか**」
を扱うのに対し、本カテゴリは「**出した後どう運用できるか**」を見る。重なる部分は記述で明示する。

機械的に検出できる範囲（lint / 型 / SAST / 監視ルールの構文）はその環境のツールに委ね、
ここでは「**人間の運用判断が必要な観点**」を扱う。

## 観点構成
- A. 失敗を前提とした I/O 設計（タイムアウト / リトライ / サーキットブレーカ / バックプレッシャ）
- B. 冪等性と再実行安全性
- C. データ整合性（同期・非同期境界での扱い）
- D. グレースフルデグラデーションとフォールバック
- E. 障害分離（バルクヘッド / 隔離 / ノイジーネイバー対策）
- F. ヘルスチェック・ライフサイクル
- G. 観測性（構造化ログ・メトリクス・分散トレース）
- H. アラート・ランブック・SLO 意識

主な参照: Google SRE Book / SRE Workbook、Michael Nygard *Release It!*、AWS Well-Architected Reliability Pillar、
The Twelve-Factor App、OpenTelemetry。

## A. 失敗を前提とした I/O 設計

ネットワーク・DB・外部 API・キュー・ファイルシステムは**いつでも失敗・遅延し得る**。
差分が外部 I/O を新設・変更している場合、失敗時の挙動が**設計**として組み込まれているかを見る。

### A-1. タイムアウト
- **すべての外部呼び出し**（HTTP / gRPC / DB / キャッシュ / キュー / 外部 SaaS）に**明示的な
  タイムアウト**が設定されているか。SDK / クライアント既定値（無制限・30 秒等）を**意識せず**に使っていないか。
- タイムアウトの値が**呼び出し元のタイムアウト**より短いか（呼び出し元が先にタイムアウトすると、
  呼び出し先の応答が捨てられて副作用だけ残るリスク）。
- リクエストのタイムアウトと**コネクションタイムアウト**を区別して設定しているか。

### A-2. リトライ戦略
- リトライが**冪等な操作**にのみ適用されているか（B 節と連動）。非冪等な書き込みを盲目的にリトライして
  二重実行を起こしていないか。
- リトライ条件が**回復可能な失敗**（5xx / ネットワーク切断 / レートリミット）に絞られ、
  **回復不能な失敗**（4xx の検証エラー / 認証エラー）でリトライしていないか。
- **指数バックオフ + ジッタ（full / decorrelated jitter）** が入っているか。固定間隔リトライで
  thundering herd を起こしていないか。
- **最大試行回数 / 最大経過時間 / バジェット**が定義されているか。無限リトライ・スタックの
  深さに依存した再帰リトライになっていないか。

### A-3. サーキットブレーカ
- 連続失敗する依存先を一時的に切り離す機構（サーキットブレーカ・bulkhead）が、**致命度の高い依存**
  （決済 / 認証 / 主要 DB 以外の補助 API 等）に適用されているか。
- 開いた回路の**半開状態（half-open）**での試験再開戦略があるか。
- 業務上、依存先が落ちている間の**代替挙動**（D 節）が決まっているか。

### A-4. バックプレッシャと過負荷制御
- 受信側の処理能力を超える流量が来たときに、**キュー無限積み上げ**で OOM や latency 爆発に
  繋がらない設計になっているか（キュー上限 + 拒否 / 古いタスク捨て / レートリミット）。
- 同時実行上限（concurrency limit / max connections / worker pool size）が**設定可能**で、
  デフォルト値が現実的か。

### A-5. レート制限・スロットリング
- 外部 API の**レート制限**を尊重しているか（429 / `Retry-After` ヘッダの考慮）。
- 自分が呼ぶ側でレート制限を**自発的にも掛けている**か（クライアント側スロットラ）。
- 自分が公開する API のレート制限は `security` I-1 / `infrastructure` で扱う。本節は依存先への
  影響に絞る。

## B. 冪等性と再実行安全性

`functionality` F 節が「**仕様として冪等を要求しているか**」を見るのに対し、本節は
「**実装メカニズムとして再実行に耐える構造になっているか**」を見る。

### B-1. at-least-once 配信前提の冪等化
- メッセージキュー（SQS / Cloud Tasks / Kafka / Sidekiq 等）からの consumer は **at-least-once 配信**を
  前提に冪等化されているか。
- 冪等化の手段（**dedupe key / idempotency key / 状態遷移ガード / UPSERT / 楽観ロック**）が
  選択されており、その選択根拠が説明できるか。

### B-2. Idempotency-Key
- HTTP API 側で**リトライ想定の作成 / 副作用系**に `Idempotency-Key` 相当の機構があるか
  （`backend` A-7 と連動）。
- キーの保存期間・衝突時挙動・本文不一致時の扱いが定義されているか。

### B-3. 状態遷移ガード
- 状態遷移を持つリソース（注文・予約・ジョブ）で、再実行時に「既に進んだ状態」を観測したら
  **何もしない / 結果を返す**ようになっているか。**取り消し済みのものを再度確定**してしまわないか。

### B-4. 部分的副作用への配慮
- 1 操作で複数の副作用（DB 書き込み + 外部 API + 通知）がある場合、**途中で失敗**した時に
  再実行しても矛盾が出ないか:
  - DB を Outbox にしてから外部送信する
  - 外部送信を冪等 API に依頼する
  - 補正処理（C-3）に切り替える
  のいずれかが選ばれているか。

## C. データ整合性（同期・非同期境界）

### C-1. トランザクション境界の妥当性
- 1 つの業務不変条件を守るための更新が**同一トランザクション**に収まっているか
  （`database` E-1 と連動）。逆に、トランザクションが過大（外部 API / 長時間処理を含む）になっていないか。
- 「DB 書き込み + 外部 I/O」を 1 トランザクションに含めると、外部 I/O 失敗時にロックが長時間保持される /
  逆に DB コミット後に外部呼び出しが失敗して整合性が崩れる、のどちらかが起きる。設計として
  どちらの側を選んだかが妥当か。

### C-2. 結果整合性の許容範囲
- 「**今すぐ整合**しなくてよいか」「**いつまでに整合**すればよいか」が要件と一致しているか。
  読み出し専用レプリカ / キャッシュからの古いデータ提示が、業務上問題ない場面に限られているか。
- ユーザー操作直後の表示で、自分の書き込み結果が**読み戻せるか**（read-your-own-writes）。

### C-3. 補正処理（Saga / Compensating Transactions）
- DB を跨ぐ整合性が必要に見える場面で、**2PC を避ける**代替（Saga / Outbox / 結果整合性 + 補償処理）が
  検討されているか（`database` E-4 と連動）。
- 採用された方式の**失敗ケース**と再実行時の冪等性が説明できるか。
- 補正に失敗したときの**最後の砦**（DLQ / 手動オペレーション手順）が明示されているか。

### C-4. 重複検知
- 「同じ操作を 2 回受け付けたら 1 回扱い」の検知点が**業務として正しい層**に置かれているか
  （API 層 / ドメイン層 / DB UNIQUE 制約 / メッセージ dedupe）。
- 検知失敗時の挙動（reject / merge / latest-wins）が要件と整合しているか。

## D. グレースフルデグラデーションとフォールバック

依存先や自分の一部機能が落ちても、**コア機能だけでも動かし続ける**設計になっているか。

### D-1. 機能の優先度づけ
- 機能が**コア / 補助 / オプション**で区別されているか。補助機能（推薦 / 類似商品 / レビュー一覧 等）の
  失敗が、コア機能（購入 / ログイン / 決済）を巻き込んでいないか。

### D-2. フォールバック
- 依存先が落ちたとき、**代替挙動**（キャッシュからの古いデータ提示 / 機能を非表示 / 既定値で進める）が
  設計されているか。フォールバックが**安全側に倒れる**こと（`robust-safety` C / `security` K-3 と連動）。
- フォールバックそのものが**監視される**ようになっているか（D-3）。

### D-3. フォールバック検知
- フォールバックパスを通った頻度・期間が**メトリクス**として可観測か（G 節）。
  「フォールバックで動いていることが誰にも気づかれず、本依存が長期間死んでいた」状態を防げるか。

### D-4. ロード時の段階的縮退
- 高負荷時に**機能を意図的にオフ**にできる仕組み（feature flag / シェディング）があるか。
- 重要操作（チェックアウト / ログイン）を保つために、軽い操作（一覧取得 / 検索）を一時遮断する
  優先度設計があるか。

## E. 障害分離（バルクヘッド・隔離）

ある一部分の障害が**全体に波及**しない構造になっているか。

### E-1. リソース分離
- 重要な処理 / 重くない処理 / 外部 API 呼び出し / バッチ系で、**スレッドプール / コネクションプール /
  ワーカープロセス**が分離されているか。1 つの遅い API がプール全体を埋めて他処理を止めていないか。

### E-2. テナント / 顧客分離
- マルチテナント環境で、**ノイジーネイバー**（特定テナントの大量リクエストが他テナントを巻き込む）
  対策があるか（テナント単位レート制限 / 重要テナントの専用キュー）。

### E-3. デプロイ単位での分離
- 1 サービスの障害で**他サービス**が連鎖落ちしない構造か。同期呼び出しチェーンが深くないか。
- 同期 RPC で繋がっている箇所が、非同期化 / イベント駆動化で切り離せる候補ではないか
  （ただし非同期化は複雑性も増えるので、本節では**現在の設計が想定障害に耐える**かを見る）。

### E-4. データ分離
- 1 つの DB 障害で**全機能**が落ちる構造になっていないか。重要機能と補助機能の DB が
  分離可能か / 分離されているか。

## F. ヘルスチェック・ライフサイクル

### F-1. ヘルスチェックエンドポイント
- **liveness**（プロセスが生きているか）と **readiness**（リクエストを受け付けられるか）が
  別エンドポイントとして定義されているか（k8s や ALB の前提）。
- readiness は、依存先（DB / キャッシュ / 外部 API）が**死んでいるとき何を返すか**が業務要件と
  整合しているか（過剰判定で全部 unhealthy にして連鎖障害を起こしていないか / 逆に依存先が死んでも
  ready を返して上流から流れ続けていないか）。

### F-2. グレースフルシャットダウン
- SIGTERM 等のシグナルを受けたとき、**処理中のリクエスト / ジョブを完了**してから終了するか。
- in-flight の HTTP リクエスト・キュー consumer・バックグラウンドタスクが、いきなり kill されると
  整合性を壊す経路がないか。
- ロードバランサ / k8s から外される **drain 期間** が設定されているか。

### F-3. 起動順序・依存準備
- 起動時に依存先が未準備（DB マイグレーション未完了等）の状態で受け付け始めて落ちないか。
- 依存先準備の**確認 → readiness OK** の流れが組まれているか。

### F-4. 構成リロード・シークレットローテーション
- 設定 / シークレットの更新がプロセス再起動なしで反映できる構造か（必要な場合）。
- シークレットローテーション時に、**旧鍵と新鍵が一時並走**できるかが設計されているか
  （`security` D-2 と連動）。

## G. 観測性（ログ・メトリクス・分散トレース）

「**本番で何が起きているか**を**事後に**追えるか」「**異常を即時に**検知できるか」を担保する。

### G-1. ログ（構造化・相関）
- ログが**構造化**（JSON / key-value）されているか。grep だけが頼りの自由文ログになっていないか。
- **相関 ID**（request id / trace id / correlation id）がリクエスト経路全体で**伝搬**しているか。
  非同期境界（キュー / バックグラウンドジョブ）でも引き継がれているか。
- **ログレベル**の使い分けが整理されているか（debug / info / warn / error / fatal の意味づけと
  本番閾値）。本番で `info` が大量に出てコストを圧迫していないか。
- **PII / 秘密情報のマスキング**が掛かっているか（`security` F-2 と連動）。
- 例外スタックトレースが、原因情報を保ったまま記録されているか（`robust-safety` H-2 と連動）。

### G-2. メトリクス（USE / RED / SLI）
- **RED** メソッド: Rate（リクエスト数）/ Errors（エラー率）/ Duration（レイテンシ分布）が
  主要エンドポイント・主要処理に揃っているか。
- **USE** メソッド: Utilization / Saturation / Errors が主要リソースで取れているか
  （CPU / メモリ / ディスク / コネクションプール / キュー深さ）。
- **業務メトリクス**: 業務上重要な値（成約数 / 失敗率 / 待機時間）が、技術メトリクスと別途取れているか。
  業務的な異常（決済成功率の急落）を技術メトリクスだけで気付けない経路がないか。
- **カーディナリティ**: ラベル / タグの値が user-id 等で爆発していないか（コスト・性能）。

### G-3. 分散トレース
- 主要なリクエスト経路に**分散トレース**（OpenTelemetry / Jaeger / Datadog APM 等）が通っているか。
- 非同期境界（キュー / イベント）でも**親 span から子 span が繋がる**ように propagation されているか。
- DB クエリ・外部 API 呼び出しが**子 span**として可視化されているか。
- 重要な業務イベント（注文 ID 等）が**span 属性**として乗っているか。

### G-4. デバッグ性
- 本番で再現性が低いバグを**事後**に追えるか:
  - リクエスト・レスポンスの**保存**（必要に応じて、PII 配慮の上）
  - 失敗時の**入力スナップショット**
  - **イベント順序**を時系列で再構成できるか
- 本番のみで起きる事象に対して、**stg / canary** で再現するための観測点が整備されているか。

### G-5. ログ・メトリクスの「足りない」検出
- 新しい分岐 / 新しい例外パスが**ログ / メトリクスに記録されない**まま投入されていないか。
  「沈黙の失敗」（silent failure）が起きないように、try-catch の握り潰し箇所に**カウンタ / ログ**が
  入っているか（`robust-safety` C-2 と連動）。

## H. アラート・ランブック・SLO 意識

OWASP A09:2025 が「Logging & **Alerting** Failures」と改名された通り、
**通知して担当が動けるか**まで含めて評価する。

### H-1. アラートの存在と質
- 重要なエラーパス・SLO 違反・依存先死活が**アラートに繋がっている**か。
  ログ / メトリクスは出るがアラートが無い、状態になっていないか。
- アラートが**actionable**（担当が次に何をすべきかが明確）か。ノイジーアラートが大量発火して
  本物が埋もれる構造になっていないか。

### H-2. ランブック・対応手順
- 新しいアラートに対する**ランブック / 対応手順**が、別ドキュメント or PR 説明にあるか。
  「鳴ったとき何をすればいいかオンコールが分からない」状態を新規導入していないか。

### H-3. SLO・エラーバジェット
- 主要機能の SLO（可用性 / レイテンシ）が定義されている場合、本変更が**エラーバジェット消費**を
  増やす方向か / 減らす方向かが意識されているか。
- リスクの高い変更を**バジェット残量がない**タイミングで投入していないか
  （ただし SLO の運用文化があるプロジェクトに限る）。

### H-4. 障害事例からの学び
- 過去のインシデント / ポストモーテムで指摘された再発防止項目に該当する変更で、
  防止策が組み込まれているか（リポジトリ内に該当ドキュメントがある場合）。

### H-5. デプロイ・ロールバック視点
- 本変更がもし**問題を起こした**ときに、安全に戻せる経路があるか:
  - **コードリバート**で戻せるか / マイグレーション・データ変更が一方通行になっていないか
  - **feature flag** で挙動を切り戻せるか
  - 戻すまでの**所要時間**が許容範囲か
- 詳細は `_reviewer-stance.md` B-4 と連動。本節は「**運用視点でロールバック可能性を見ている**」点を
  ここで明示する。

## 共通: severity 判定の目安（本カテゴリ固有の補足）

`_reviewer-stance.md` A-3 を基本に、本カテゴリでは以下を加味する:

| 区分 | severity の目安 |
|------|----------------|
| 本番で障害発生時に**回復不能 / 検知不能 / データ消失**に直結する欠落（タイムアウト無し・冪等性無し・ログ無し等で気付けない） | blocker |
| 本番運用上の重大リスクだが連鎖条件あり / 観測性の中核要素が欠けている | major |
| 観測性 / 縮退設計の改善余地、現状の障害シナリオでは大きく困らない | minor |
| アラート文言・命名・好みの範囲 | nit |

- **「本番で起きると想定される具体シナリオ」が描けるかどうか**で severity を分ける。
  「タイムアウト未設定」は典型的な blocker / major（外部依存の遅延で連鎖障害になり得る）。
  「ログにユーザー ID が含まれていない」は通常 minor だが、認証障害切り分けに必須なら major。
- 本番未運用のサービス / 内部ツール等で運用負荷が低い場合は、severity を下げてよい。

## 除外・スコープ外

- **コードレベルの不正状態回避（型・並行・例外設計）** — `robust-safety` カテゴリ
- **段階的出荷戦略（Expand & Contract / マイグレーション分割 / ロールアウト）** — `_reviewer-stance.md` B 節
  （本カテゴリは出した後の運用挙動・ロールバック経路に絞る）
- **DB スキーマ・マイグレーションの安全性そのもの** — `database` D 節
- **API 設計の HTTP セマンティクス** — `backend` A 節
- **CI/CD パイプライン・IaC 設計そのもの** — `infrastructure` カテゴリ
- **セキュリティ目的のロギング（監査ログ・改竄耐性）** — `security` J 節
  （本カテゴリは運用観点で「異常を追えるか」「sandbox で再現できるか」に絞る）
- **計算量・スループット・レイテンシ最適化** — `performance` カテゴリ
- **要求仕様の充足そのもの・エッジケース挙動** — `functionality` カテゴリ
  （仕様として冪等を要求しているかは `functionality` F、実装メカニズムは本カテゴリ B）
- **メトリクス可視化ダッシュボード・Grafana 設定そのもの** — レビュー対象に含まれる場合は `infrastructure`

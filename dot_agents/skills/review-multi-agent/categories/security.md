# security 観点（優先順位: 1）

## このカテゴリで見るもの
機密性・完全性・認証/認可・入力検証・暗号・サプライチェーン・秘密情報・セキュリティ運用。
**OWASP Top 10:2025**（A01〜A10）と **OWASP API Security Top 10:2023**（API1〜API10）を骨格にし、
**OWASP ASVS 5.0** と **OWASP Cheat Sheet Series** を必要に応じて参照する。

機械的に検出できるルール（lint / SAST / 依存スキャナ / シークレットスキャナ等）はその環境のツールに委ね、
ここでは「**人間の判断が必要なセキュリティ上のトレードオフ**」を見る。攻撃者視点で「この差分が本番に
入った瞬間に悪用可能になる経路があるか」「想定外の入力・状態でも安全側に倒れるか」を中心に評価する。

## 観点構成
- A. アクセス制御（認可） — OWASP A01:2025（SSRF を含む）/ API1, API3, API5
- B. 認証・セッション — OWASP A07:2025 / API2
- C. インジェクション — OWASP A05:2025
- D. 暗号・鍵・乱数 — OWASP A04:2025
- E. セキュリティ設定・ヘッダ・CORS — OWASP A02:2025 / API8
- F. 機密情報・PII・秘密情報の取り扱い — OWASP A04:2025 / A08:2025 / A09:2025
- G. データ整合性・改竄耐性 — OWASP A08:2025
- H. サプライチェーン — OWASP A03:2025
- I. リソース濫用・ビジネスロジック濫用 — OWASP API4 / API6
- J. セキュリティロギング・アラート — OWASP A09:2025
- K. Insecure Design（横断的な設計欠陥） — OWASP A06:2025

> **注**: OWASP Top 10:2025 は 2025-11 公開の Release Candidate を前提にしている。
> 公式仕様の細部は将来更新され得るため、確実な引用が必要なときは公式ページ（参考資料末尾）の最新版を確認する。

## A. アクセス制御（認可）

OWASP Top 10:2025 で **A01: Broken Access Control** は引き続き 1 位。2025 改訂で **SSRF** が A01 に統合された。
**認可は本カテゴリの最重要観点**で、blocker / major 判定が最も多くなる領域。

### A-1. オブジェクトレベル認可（BOLA / IDOR）— API1
- リソース ID をパラメータで受け取るエンドポイントで、**呼び出し元がそのリソースに対する権限を持つか**が
  ハンドラ内で検証されているか。「ログインさえしていれば任意の `userId` / `orderId` を渡せる」状態に
  なっていないか。
- 検証が **データ取得後の if 文** ではなく、**取得クエリの条件**（`WHERE owner_id = $current_user`）に
  組み込まれているか（取得してから捨てる方式は性能・複雑性の両面で漏れやすい）。
- 推測可能な連番 ID を外部に晒している場合、**enumeration による情報漏洩**（404 と 403 の差分など）が
  起きていないか。UUID / ULID への移行や 403 統一が検討されているか。

### A-2. 機能レベル認可 — API5
- ロール / 権限が必要なエンドポイントに **認可ミドルウェア / ガードが実装され、漏れなく適用** されているか。
  ルート定義側で「デフォルト要認証 + 例外で公開」の方針が取れているか（許可リスト型より拒否リスト型は漏れやすい）。
- 管理機能 / 内部 API が**パス推測**（`/admin/...` `/internal/...`）だけで保護されていないか。
- 新規エンドポイント追加時、認可テストが追従しているか（テストの不在は major 以上の指摘候補）。

### A-3. プロパティレベル認可 — API3（Mass Assignment / Excessive Data Exposure）
- リクエスト DTO が **書き込み可能なフィールドを明示** しているか。クライアントが渡した JSON を
  そのままモデルに `Object.assign` / `update_attributes` する **mass assignment** になっていないか。
- レスポンス DTO が **読み取り可能なフィールドだけを返す** 設計になっているか。エンティティをそのまま
  シリアライズして、内部 ID・パスワードハッシュ・他テナントの参照が漏れていないか。
- GraphQL / 動的フィールド選択がある場合、未認可ユーザーが選択可能なフィールドが制御できているか。

### A-4. 認可チェックの一元化と Default Deny
- 認可ロジックが**一箇所に集約**され、各ハンドラで重複定義されていないか。重複している場合、
  ロジックがドリフトしていないか。
- **Default Deny**: 認可情報がない場合は許可ではなく拒否に倒れる設計か。
  例: ポリシー未定義のリソースに対して「ポリシーが見つからない → allow」の経路が存在しないか。

### A-5. 権限昇格・横移動
- 自分のロールを変更できる経路（`PATCH /me { role: "admin" }`）が遮断されているか。
- 他テナント / 他組織のリソースへの **横移動**（tenant id を URL から受け取って素通し）になっていないか。
  マルチテナントでは **tenant_id をすべてのクエリ・全 UNIQUE 制約に含める** 方針と整合しているか
  （`database` カテゴリ G-3 と連動）。

### A-6. SSRF（A01:2025 へ統合）— API7
- サーバが **外部 URL をクライアント入力から受け取って fetch する** 経路で:
  - allowlist 方式（許可ドメインのみ）か。block-list（`127.0.0.1` を弾く等）の網羅は不十分（IPv6・短縮 IP・DNS rebinding で迂回されやすい）
  - **メタデータエンドポイント**（`169.254.169.254` / クラウド IMDS / k8s API）への到達が遮断されているか
  - リダイレクトの追跡で **再度の解決と検証**が走っているか
- 内部 DNS 名・内部 IP が解決される経路が、ネットワーク層でも遮断されているか。

### A-7. ビジネスフローの濫用 — API6
- 「**正規の API を機械的に大量実行**することで害が出る」フロー（ポイント交換・抽選参加・在庫予約 等）が、
  レート制限・device fingerprint・CAPTCHA・abuse 検知のいずれかで保護されているか。
  この観点は I-2 と連動。

## B. 認証・セッション

OWASP Top 10:2025 **A07: Authentication Failures** / API2 に対応。

### B-1. パスワード保管
- パスワード保管に **メモリ / CPU ハード関数**（Argon2id / scrypt / bcrypt）が使われているか。
  `MD5` / `SHA-1` / `SHA-256` 単発のハッシュになっていないか。
- パラメータ（cost / iterations / memory）が **現代の推奨値** に近いか
  （ASVS 5.0 / Cheat Sheet "Password Storage" の推奨を確認）。
- パスワード比較が **定数時間比較**（`crypto.timingSafeEqual` / `hmac.compare_digest` / `subtle.compare`）か。

### B-2. トークン・セッション管理
- セッション ID / リフレッシュトークンが **CSPRNG** で 128 bit 以上か（C-3 / D-3 と連動）。
- Cookie の `Secure` / `HttpOnly` / `SameSite` 属性が必要に応じて付与されているか
  （詳細フラグは E-4）。
- **JWT / 自前トークン**:
  - `alg: none` の受理を防いでいるか、`HS` と `RS` の混用で公開鍵を秘密鍵として誤検証していないか
  - `exp` / `nbf` / `iss` / `aud` を検証しているか
  - **失効（revocation）** が必要な業務要件に対して、ステートレス JWT で済ませて取り消せない設計に
    なっていないか（短寿命化 + リフレッシュ + ブラックリストの選択）
- セッション固定化（fixation）対策: ログイン直後に **セッション ID が再発行** されているか。
- ログアウトでサーバ側のセッション / リフレッシュトークンが **確実に無効化** されているか。

### B-3. 多要素認証（MFA）
- 高権限操作（管理画面ログイン / 重要設定変更 / 高額決済）に MFA / step-up 認証が要求されているか。
- TOTP / WebAuthn / プッシュ通知のいずれかで、**SMS のみに依存** していないか
  （SIM swap / SS7 攻撃の歴史的事例）。
- リカバリーコード生成・配布・再生成のフローが、本人確認・回数制限を伴っているか。

### B-4. アカウント回復経路
- パスワードリセット・メール変更・電話番号変更などの**回復経路が、ログインそのものよりも弱い認証**で
  通っていないか。回復が認証強度の最弱リンクになることが多い。
- リセットトークンが **短寿命**・**ワンタイム**・**第三者に推測不可能** か。
  メール内リンクの URL が full URL ではなく、事前に発行したトークン参照になっているか。
- 「セキュリティ質問」が、SNS で公開済みの情報で回答可能になっていないか。

### B-5. ブルートフォース・credential stuffing 対策
- ログイン・MFA 検証・パスワードリセットに **レート制限 + ロックアウト** が組まれているか。
  IP 単位のみだと NAT / 共有環境で誤遮断・ボットネットで貫通されやすい。アカウント単位 + IP 単位の
  併用、または失敗試行のスコアリング型が検討されているか。
- credential stuffing 対策として、**漏洩済みパスワードリスト**（HIBP API / k-anonymity 経由）との
  照合や、異常ログイン検出（地理 / デバイス）が業務要件に応じて検討されているか。

## C. インジェクション

OWASP Top 10:2025 **A05: Injection** に対応。XSS は 2021 で Injection に統合され、2025 でも同枠。

### C-1. SQL / NoSQL / ORM
- パラメータ化クエリ / プリペアドステートメント / ORM のクエリビルダで構築されているか。
  **文字列連結** で SQL を作っている箇所はないか（テーブル名・列名の動的注入は allowlist 化）。
- ORM の `raw` / `query` メソッドや `Sequel.lit` 相当に、未検証の入力が渡っていないか。
- NoSQL（MongoDB の `$where` / `$regex`）に **演算子注入**（`{ "$ne": "" }` など）が成立しないか。

### C-2. OS コマンド・シェル
- `exec` / `system` / `child_process.exec` などで **シェルを介して** ユーザー入力を渡していないか。
  `child_process.spawn(cmd, [args])` のように **引数配列**で渡し、シェル展開を回避しているか。
- `bash -c "..."` パターンで `${var}` / `$()` がエスケープなしに埋め込まれていないか。

### C-3. XSS / HTML / DOM
- テンプレートエンジンの **自動エスケープ** が有効か。`unsafe-html` / `dangerouslySetInnerHTML` /
  `v-html` / `bypassSecurityTrust*` の使用箇所が、信頼済み入力にのみ限定され、根拠が説明できるか。
- リッチテキストを許容する経路で、サニタイザ（DOMPurify / `sanitize-html` 等）が **allowlist 構成** で
  使われているか。タグ / 属性 / URL スキーマ（`javascript:` / `data:`）の制限が妥当か。
- **コンテキスト別エンコード**: HTML 本文 / 属性値 / URL / JS リテラル / CSS で、適切なエンコードが
  選ばれているか（一律 HTML エスケープでは不足するケースあり）。
- DOM ベース XSS: `innerHTML` / `document.write` / `eval` / `new Function` / `setTimeout(string, ...)` に
  ユーザー入力が流れ込んでいないか。

### C-4. テンプレートインジェクション（SSTI）
- サーバ側テンプレート（Jinja / ERB / Twig / Handlebars / Liquid 等）の **テンプレート文字列自体** に
  ユーザー入力を結合していないか（変数渡しではなく）。
- LLM プロンプトテンプレートにユーザー入力を結合する場合、後段で**プロンプトインジェクションの被害**
  （権限のあるツール呼び出しを誘発）が起きないかが、設計されているか。

### C-5. ヘッダ・ログ・LDAP / XPath / 正規表現
- HTTP レスポンスヘッダ / リダイレクト URL / ログ出力に、改行コード（CR/LF）を含む入力が
  そのまま埋め込まれていないか（**CRLF Injection** / Log Forging）。
- LDAP / XPath クエリの組み立てに、エスケープなしの入力が混ざっていないか。
- ユーザー入力から**正規表現**を組み立てている経路で、**ReDoS**（catastrophic backtracking）の評価が
  されているか（特に `(a+)+`, `(a|aa)+` 系の重ねがけ）。

### C-6. デシリアライゼーション
- 信頼できない入力を `pickle` / `Marshal` / Java `ObjectInputStream` / `unserialize` などで
  デシリアライズしていないか。**任意コード実行**を許す経路がないか。
- JSON / YAML でも、**型タグ**（YAML `!!python/object` / Jackson polymorphic typing）で
  クラスインスタンス化が走らないように制限されているか。

## D. 暗号・鍵・乱数

OWASP Top 10:2025 **A04: Cryptographic Failures** に対応。

### D-1. アルゴリズム選定
- ハッシュ / 共通鍵暗号 / 公開鍵暗号 / 鍵交換 / 署名で、**現代水準のアルゴリズム** が選ばれているか:
  - パスワードハッシュ: Argon2id / scrypt / bcrypt（B-1 と連動）
  - 汎用ハッシュ: SHA-256 以上（MD5 / SHA-1 を完全性目的で使っていないか）
  - 共通鍵: AES-GCM / ChaCha20-Poly1305 など **AEAD**。CBC + 自前 HMAC は誤りやすい
  - 公開鍵: 2048 bit RSA 以上 / Ed25519 / ECDSA P-256 以上
- アルゴリズムを**自作・改造**していないか（カスタム XOR / 自前パディング 等）。

### D-2. 鍵管理
- 鍵が**コードリポジトリ・コンテナイメージ・ログ**に含まれていないか（F-1 と連動）。
- 鍵が **KMS / Secret Manager / HSM** などの専用基盤で管理されているか。アプリ内で平文保持する場合、
  メモリスキャンへの耐性・寿命が許容範囲か。
- **鍵ローテーション**の手順が用意され、運用上実行可能か。鍵 ID / バージョンが暗号文に
  埋め込まれているか（鍵切り替え時の旧データ復号経路）。
- 鍵削除（暗号削除 / crypto-shredding）が、**論理削除の代替**として要件と整合するか。

### D-3. 乱数（CSPRNG）
- セキュリティ用途の乱数（トークン / セッション ID / nonce / IV / ソルト）に
  **CSPRNG**（`crypto.randomBytes` / `secrets.token_bytes` / `crypto/rand` / `getrandom`）が
  使われているか。`Math.random` / `rand()` / `Random` クラスの暗号目的での使用がないか。
- nonce / IV の **再利用** が起きないか（特に AES-GCM の nonce 再利用は致命的）。

### D-4. TLS / 通信暗号
- HTTPS が必須化され、HTTP からのリダイレクトで **HSTS** が効いているか（E-2 と連動）。
- 内部通信（サービス間 / DB / キャッシュ）が TLS 化されているか、TLS 不要と判断したなら根拠が説明できるか。
- 外部 API クライアント / mTLS で、**証明書検証を無効化**していないか（`verify=False` / `rejectUnauthorized: false`）。
  自己署名証明書を扱う場合、信頼ストアへの追加で対応しているか。
- 古い TLS バージョン / 弱い cipher suite が許可されていないか（TLS 1.2 以上が現代水準）。

### D-5. データ保護
- **at rest**: DB / バックアップ / ファイルストレージで、必要箇所が暗号化されているか。
  カラムレベル暗号 / アプリ層暗号の必要性（管理者からの保護 / multi-tenancy の隔離）が評価されているか。
- **in transit**: D-4 と連動。
- パスワードを暗号化（reversible）してはいけない経路（一方向ハッシュにすべき）で、暗号化を選んでいないか。

## E. セキュリティ設定・ヘッダ・CORS

OWASP Top 10:2025 **A02: Security Misconfiguration**（2021 の 5 位から 2025 で 2 位に上昇）/ API8 に対応。

### E-1. デフォルトの安全側固定
- **本番モード**でデバッグ機能（スタックトレース表示 / 管理コンソール / プロファイラ / 開発用 CORS 緩和）が
  オフになっているか。環境差（dev/stg/prod）の切替が、設定ファイル / 環境変数で**確実**に分岐しているか。
- フレームワークのデフォルト設定で、**安全でない既定**（CSRF オフ / 強制 HTTPS オフ等）が残っていないか。

### E-2. セキュリティヘッダ
公開 Web では以下が想定通りに付与されているか:
- **Content-Security-Policy (CSP)**: `default-src 'self'` ベース。`unsafe-inline` / `unsafe-eval` /
  ワイルドカードホストの許容根拠が説明できるか。`nonce` / `hash` ベースの inline 許可が選択されているか
- **Strict-Transport-Security (HSTS)**: `max-age` が現代水準（少なくとも 6 ヶ月以上）か、
  `includeSubDomains` / `preload` の方針が明確か
- **X-Content-Type-Options: nosniff**
- **Referrer-Policy**: 内部 URL を外部に流さない方針か（`strict-origin-when-cross-origin` 程度）
- **X-Frame-Options** / CSP `frame-ancestors`: クリックジャッキング対策
- **Permissions-Policy** / **Cross-Origin-{Opener,Embedder,Resource}-Policy**: 必要に応じて

### E-3. CORS
- `Access-Control-Allow-Origin: *` を **クレデンシャル付き API** に適用していないか。
- Origin の動的反映で、**許可リスト方式**になっているか（`origin === req.headers.origin` 全反射は危険）。
- プリフライト（OPTIONS）の `Allow-Methods` / `Allow-Headers` が必要最小限か。

### E-4. Cookie 属性
- セッション Cookie / 認証 Cookie に `Secure` / `HttpOnly` / `SameSite=Lax|Strict` が付与されているか。
- クロスサイト送信が必要な経路で `SameSite=None` を使う場合、`Secure` 必須かつリスク評価されているか。
- `Domain` / `Path` の指定が広すぎず、不要なサブドメイン共有になっていないか。

### E-5. 不要機能・公開面の最小化
- 使っていないエンドポイント・管理 API・デバッグルート・古い API バージョンが**閉じている**か（API9 / Improper Inventory Management）。
- 自動生成された OpenAPI / GraphQL Introspection / `/actuator` / `/debug/pprof` 等が、本番で
  認可なしに露出していないか。
- ディレクトリリスティング・サーバ署名（`Server: ...`）・スタックトレース漏洩が抑止されているか。

## F. 機密情報・PII・秘密情報の取り扱い

OWASP Top 10:2025 **A04 / A08 / A09** にまたがる横断観点。`database` カテゴリ I 節 / `reliability` の
観測性とも連動するが、**漏洩観点から見るのはここ**。

### F-1. 秘密情報のハードコード・コミット
- API キー / DB 接続文字列 / 署名鍵 / OAuth client secret が**ソースコード・テストフィクスチャ・
  コミット履歴**に残っていないか。`.env.example` がプレースホルダのみか。
- コンテナイメージ / ビルド成果物に秘密情報が焼き込まれていないか。

### F-2. ログ・監視・トレース
- 機微情報（パスワード・トークン・カード番号・PII・メール本文）が**ログに出力されていない**か。
  リクエスト / レスポンスをそのまま dump するロガーで素通しになっていないか。
- 構造化ログのフィールド単位で **redaction**（マスキング）が定義されているか。
- 監視 / APM / トレース（OpenTelemetry / Datadog 等）に対しても同等の方針が適用されているか。

### F-3. エラーレスポンス・スタックトレース漏洩
- 本番のエラーレスポンスで、内部実装（クラス名 / SQL / 内部パス / 環境変数）が漏れていないか。
- 認証失敗・存在チェックで、「ユーザーが存在するか」「パスワードが正しいか」を区別できる
  メッセージを返していないか（enumeration 対策）。

### F-4. バックアップ・分析基盤・派生データ
- 本番 DB のバックアップ / DWH 連携 / 分析基盤に、PII / 機微情報がそのまま流れていないか。
  必要に応じて匿名化 / 集計化されているか。
- ステージング・開発環境に **本番データのコピー**が入る場合、マスキング / 合成データ化されているか。

### F-5. 削除・保持期間
- ユーザーの削除権利（GDPR / CCPA 相当）に対応する**物理削除パス**が設計されているか
  （`database` F-2 と連動）。
- 監査ログ・バックアップ・派生データに対しても削除手順があるか。

---

## G. データ整合性・改竄耐性

OWASP Top 10:2025 **A08: Data Integrity and Confidentiality Failures** に対応。

### G-1. CSRF
- 状態を変更する API が、**CSRF トークン**（同期トークン / double submit cookie）または
  `SameSite=Strict|Lax` Cookie + プリフライト要求トリガで保護されているか。
- API トークンを `Authorization` ヘッダで送る方式で、Cookie 認証と**併存**させていないか
  （Cookie 認証だけ CSRF 攻撃可能になる経路を残していないか）。

### G-2. オープンリダイレクト
- リダイレクト先 URL をクエリで受け取る経路（`?next=...`）で、**自ドメイン allowlist** で
  検証しているか。`http://example.com.evil.com` 系のサブストリング検証で誤判定していないか。

### G-3. ファイルアップロード
- アップロードされたファイルの **MIME 種別検証**（拡張子だけでなく magic byte / Content-Type 両方）が
  行われているか。
- 実行可能拡張子（`.php` / `.jsp` / `.html` / `.svg`）の保存・配信ポリシーが安全か。
  実行ディレクトリ外への保存 / `Content-Disposition: attachment` での配信。
- パストラバーサル（`../`）・絶対パス・null byte が拒否されているか。

### G-4. クライアントサイド整合性
- 外部 CDN / サードパーティ JS の読み込みで、**Subresource Integrity (SRI)** が利用できる経路で
  使われているか。
- 自社ロード経路でも、**ビルド成果物の改竄検出**（ハッシュ・署名）が必要な経路で抜けていないか。

### G-5. クライアント検証への依存
- 「JS で `disabled` / `readonly` にしているから安全」「クライアント側のバリデーションを通っている
  はず」という設計になっていないか（**サーバ側で必ず再検証**）。

---

## H. サプライチェーン

OWASP Top 10:2025 **A03: Software Supply Chain Failures**（2025 で新カテゴリ）に対応。

### H-1. 依存関係
- 直接依存 / 推移的依存に**既知脆弱性**が残っていないか（ツールでの自動検知に加え、
  選定時の活発度・メンテ状況・既知問題のトリアージが妥当か）。
- 依存追加時、**ライセンス / 配布元 / メンテナ**の確認が行われているか
  （typosquatting / 似たパッケージ名による偽パッケージ取り込み）。
- **lock 機構**（`package-lock.json` / `Cargo.lock` / `go.sum` / `Pipfile.lock`）が運用されているか。
  CI で integrity 検証（npm `--frozen-lockfile` / `pip --require-hashes`）されているか。

### H-2. ビルド・パイプラインの改竄耐性
- CI / CD のシークレット（デプロイ鍵 / レジストリ token / 署名鍵）が、必要最小権限で管理されているか。
- ビルド成果物（コンテナイメージ・パッケージ・バイナリ）が**署名**されているか
  （Sigstore / cosign / Sigstore policy / SBOM 添付）。
- 配布先（npm / PyPI / コンテナレジストリ / S3）への push 権限が、レビュー前に押し込めない構造か。

### H-3. ベースイメージ・ランタイム
- コンテナのベースイメージが、**信頼できる発行元**かつ**意図したダイジェスト**で固定されているか
  （`:latest` 参照は特定性がない）。
- 言語ランタイム / OS パッケージのバージョンが、サポート範囲・脆弱性更新サイクルと整合しているか。

### H-4. AI / モデル / 学習データのサプライチェーン
- 第三者から取得した**モデル重み・データセット・プロンプト**を本番に組み込む場合、
  改竄・有害コンテンツ・ライセンス汚染の検証が行われているか。
- LLM が外部 MCP / ツールを呼ぶ経路で、ツール定義・スキーマが信頼できる発行元から来ているか。

---

## I. リソース濫用・ビジネスロジック濫用

OWASP API4: Unrestricted Resource Consumption / API6: Unrestricted Access to Sensitive Business Flows。

### I-1. レート制限・スロットリング
- 公開エンドポイント・コストの高い処理（検索 / レポート生成 / メール送信 / 外部 API 呼び出し）に、
  **レート制限**が掛かっているか。アカウント単位 / IP 単位 / グローバルの組み合わせが妥当か。
- レート制限の**ストレージ**（in-memory / Redis 等）が分散環境で正しく機能するか。

### I-2. リソース消費の上限
- **アップロードサイズ** / **JSON ペイロードサイズ** / **配列要素数** / **文字列長** に上限が
  あるか。デフォルト値が緩すぎないか（巨大入力でメモリ枯渇する経路がないか）。
- **画像 / 動画変換**・**ZIP 展開**（zip bomb 対策）・**正規表現**（C-5 ReDoS）・
  **再帰深さ**などの計算量爆発が抑止されているか。
- **ページング上限**・**N+1 防止**（`backend` / `database` / `performance` カテゴリと連動）が、
  攻撃者の DoS 経路にならないように設計されているか。

### I-3. ビジネスロジック濫用 — API6
- ポイント・クーポン・抽選・在庫予約・友達招待など、**金銭的価値が動く**フローに、
  自動化耐性（CAPTCHA / device fingerprint / 行動分析 / 招待制 等）が組まれているか。
- 並行実行を悪用した二重取り（クーポン重複適用・在庫オーバー予約）が、楽観ロック / 排他制御で
  防がれているか（`database` E-3 と連動）。

### I-4. 外部 API への依存
- 外部 API 呼び出しが**タイムアウト / 接続上限**を持ち、外部の遅延がアプリ全体を巻き込まないか
  （`reliability` と連動するが、可用性低下が攻撃面になる経路はここで指摘）。
- **API10 (Unsafe Consumption of APIs)**: 外部 API のレスポンスを**信頼境界外**として扱い、
  検証なしで内部処理に流していないか（A-1 / C 系と連動）。

## J. セキュリティロギング・アラート

OWASP Top 10:2025 **A09: Security Logging & Alerting Failures**（2025 で「& Alerting」が追加され、
**通知して行動できるか**まで含まれるようになった）。

### J-1. 監査ログの網羅
- 以下のイベントに**識別可能な監査ログ**が残るか:
  - 認証成功 / 失敗 / MFA イベント / パスワード変更 / ロール変更
  - 重要リソースの作成 / 変更 / 削除
  - 認可拒否（403）
  - 管理操作・ポリシー変更
- ログに **誰が（actor）・何に（resource）・どこから（IP / UA）・いつ・結果** が含まれているか。
- 過剰収集（パスワードや機密情報の取り込み）が起きていないか（F-2 と連動）。

### J-2. ログ改竄耐性・保全
- アプリ自身が自分のログを書き換え可能な構造になっていないか（書き込み専用ストリームへの委譲）。
- 保管期間が要件と整合し、削除権利との兼ね合いが整理されているか。

### J-3. アラートとインシデント対応
- 異常イベント（連続認証失敗 / 認可拒否の急増 / 異地域ログイン / サプライチェーン警告）が
  **検知 → 通知 → 当番が動ける**経路まで設計されているか。
- 通知先が個人メール 1 つに依存していないか。**アラート疲れ**で重要通知が埋もれない閾値設計か。

---

## K. Insecure Design（横断的な設計欠陥）

OWASP Top 10:2025 **A06: Insecure Design** は、コードレベルの bug ではなく**設計時点の欠陥**を扱う。
**脅威モデリングの不在**そのものが指摘対象になり得る。

### K-1. 脅威モデリング
- 当該フロー / 機能に対して、**誰が・何を狙い・何を悪用するか**が考慮された痕跡（設計ドキュメント /
  PR 説明 / コメント）があるか。複雑な認可・課金・データ削除フローで、攻撃者ストーリーが議論されたか。
- 信頼境界（trust boundary）と、その境界での検証が図示・記述されているか。

### K-2. 安全でないデフォルトに依存した機能拡張
- 「機能を後から追加するときに、デフォルト ON / 共有設定 / 公開可視 になるパス」が新設されていないか。
  例: 新しい共有機能が「誰でもアクセス可能」を初期値にしている、新しい API がデフォルトで公開、など。

### K-3. 安全側に倒すフェイルクローズ
- 機能フラグ / 認可ポリシー / 暗号鍵が**取得できないとき**、機能停止（fail close）に倒れるか、
  許可（fail open）に倒れるか。**認可・課金・送信に関わる処理は fail close** が原則
  （`robust-safety` C-1 と内容重なるが、セキュリティ観点での確認はここで行う）。

### K-4. プライバシー・最小収集
- 取得・保存する個人情報が、**業務上必要な最小限**に絞られているか。
  「将来使うかも」で無目的に収集していないか（GDPR / APPI のデータ最小化原則）。

## 共通: severity 判定の目安（本カテゴリ固有の補足）

`_reviewer-stance.md` の A-3 を基本に、本カテゴリでは以下を加味する:

| 区分 | severity の目安 |
|------|----------------|
| 本番投入後すぐに悪用可能、認証・認可・機密漏洩・任意コード実行に直結 | blocker |
| 本番で悪用可能だが連鎖条件が必要 / 直近の影響は限定的だが放置すべきでない | major |
| 標準準拠やヘッダ整備など、現状被害が出ていない強化提案 | minor |
| 命名・スタイル・好みの範囲 | nit |

- 攻撃シナリオが**具体的に**書ける場合は severity を上げる。書けない（理論上の懸念に留まる）なら下げる。
- 同じ脆弱性カテゴリでも、**到達可能性**（公開エンドポイントか / 内部限定か / 認証必須か）で severity が変わる。

---

## 除外・スコープ外

- **機能仕様の充足そのもの** — `functionality` カテゴリ
- **コードレベルの堅牢性（型による不正状態回避・並行安全・例外設計の網羅性）** — `robust-safety` カテゴリ
  （ただしセキュリティ事故に直結する経路で具体的な悪用シナリオが描ける場合は本カテゴリで指摘）
- **OWASP A10:2025 (Mishandling of Exceptional Conditions)** の大部分 — 一次的には `robust-safety` H 節で扱う
- **可用性低下のみが論点の DoS / スループット問題** — `performance` / `reliability` カテゴリ
  （ただし**意図的に低リソースで濫用可能**なケース（API4 / I-2）は本カテゴリで扱う）
- **API 設計の HTTP セマンティクス（メソッド・ステータス・URI 形）** — `backend` A 節
- **DB スキーマ・FK・トランザクション設計の正しさそのもの** — `database` カテゴリ
- **lint / SAST / 依存スキャナ / シークレットスキャナで自動検出可能なルール** — 各プロジェクトの CI に委譲

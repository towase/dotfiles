# functionality 観点（優先順位: 2）

## このカテゴリで見るもの
**「この差分は、求められたことを正しく・抜け漏れなく実現できているか」** を見る。
要求仕様 / Issue の達成条件・受け入れ基準が満たされているかと、ハッピーパス以外の **エッジケース** /
**境界条件** / **状態組合せ** の挙動、API・型・スキーマなどの **契約**との整合性が中心。

`robust-safety` カテゴリが「コードが不正状態を作らないか」をミクロな型/並行/例外設計の観点で見るのに対し、
本カテゴリは **業務要件視点** で「**何を作ると約束したか / それが実装に反映されているか / 約束していない
ケースで暴発しないか**」を見る。両者は交差するため、**業務要件として漏れているなら本カテゴリ**、
**コード上の不変条件として担保されていないなら robust-safety** に振る。

機械的に検出できる範囲（型・lint・スキーマ整合の自動検証）はその環境のツールに委ねる。
ここでは「**人間がリポジトリ内のドキュメントや会話履歴も読まないと判断できない**」観点を扱う。

## 観点構成
- A. 要求の同定と達成状況（受け入れ基準のカバレッジ）
- B. エッジケース・境界条件
- C. 契約（API / 型 / スキーマ / イベント）との整合
- D. エラーパスと結果の表現
- E. 状態・条件の組合せ網羅
- F. 副作用・冪等性（仕様視点）
- G. テスト・受け入れ検証
- H. データの意味的正しさ

---

## A. 要求の同定と達成状況

差分が**何を達成すべきか**を、ソースコードの外（PR 説明・Issue・設計ドキュメント・会話履歴）から
特定し、**それと差分の内容が一致しているか**を見る。本節の指摘は他カテゴリの基礎になるため、
最初に行う。

### A-1. 要求の出所と特定
- PR 説明・関連 Issue・関連設計ドキュメント・直近のレビューコメントが**読み取れる範囲で参照**され、
  「何を実現する差分か」が一意に取れているか。
- 要求が **アウトカム（得たい成果）** で書かれているか、**アウトプット（具体的な実装物）** で書かれているか。
  アウトカム要件なのに「実装した = 達成」と取り違えていないか
  （プロジェクト方針: GitHub Issue は原則アウトカム記載）。
- 要求が複数経路（Issue / PR 説明 / Slack / コメント）に散らばっていて**一貫していない**場合、
  最終仕様がどれかが PR 上で確認可能か。

### A-2. 受け入れ基準のカバレッジ
- 受け入れ基準（Acceptance Criteria）が箇条書きで列挙されている場合、**各項目に対応する変更**が
  差分内に存在するか。表面的に類似した変更が混じっていて、本来の項目が**漏れている**ことはないか。
- 「すべき」「しないべき」の両面が網羅されているか。「動くようになった」の確認だけで、
  「壊してはいけないものが壊れていない」確認が抜けていないか。

### A-3. スコープと差分の整合
- **スコープ過小**: 要求に含まれる経路のうち、**未対応のもの**が差分に残っていないか
  （別 PR で扱う旨が明示されていればよい）。
- **スコープ過大**: 要求に含まれない**ついで修正**が紛れ込んでいないか。
  混在するとレビュー困難・revert 困難になる（`maintainability` と連動するが、ここでは「要求外のものが
  まぎれて意図せぬ振る舞い変更になっていないか」を見る）。
- 既存機能の挙動が**意図せず変わっていない**か。仕様変更を伴う場合、それが要求書に明記されているか。

### A-4. 暗黙の要件
- リポジトリの **`AGENTS.md` / `CLAUDE.md` / `docs/` / 既存実装パターン**から読み取れる
  「明示されていないが守るべき規約」に違反していないか。
- 既存の類似機能の挙動と**一貫している**か（命名・エラー時挙動・並び順・ページング方式等）。
  類似機能が複数あり、慣習に揺れがある場合、差分がどちらに寄せたかの判断が説明できるか。

## B. エッジケース・境界条件

「ハッピーパスでは動く」コードに対する典型的な落とし穴を、業務視点で並べる。
コード上の堅牢性（型による不正状態の不可表現化等）は `robust-safety` で扱うため、ここでは
**業務上発生しうる入力・状態の境界**で結果が要求通りになるかを見る。

### B-1. 入力の数量境界
- **0 件 / 1 件 / 多数件** で挙動が変わる箇所（リスト処理・集計・通知）で、各境界が要求通りか:
  - 空リストの集計で「ゼロ」を返すべきか「未定義」を返すべきか
  - 1 件しかないときに複数前提のメッセージ・複数前提の改ページが破綻していないか
  - 大量件数で UI / API のレスポンスが要求通りか（`performance` ではなく**仕様としての挙動**を見る）
- ページング上限・取得件数上限・上限超えのときの挙動（切り捨て / エラー / 続きを返す）が
  要求と一致しているか。

### B-2. 値の境界
- **数値**:
  - 0 / 負数 / 最小値 / 最大値 / 限界に近い値（例: `INT_MAX`、有効桁数の境界）の扱い
  - 通貨の最小単位（円なら 1、USD なら 0.01）の境界
  - 比率・割合・上限が**仕様通り**に効いているか（小数点切り捨てか四捨五入か / 上限を含むか）
- **文字列**:
  - 空文字 / 1 文字 / 仕様上の最大長 / 最大長 +1 / null
  - 前後空白 / 全角空白 / タブ / 改行 / 絵文字 / サロゲートペアの扱いが業務要件と一致しているか
  - 大文字小文字・全角半角の正規化が比較・検索・保存で一貫しているか
- **真偽値・null・undefined**:
  - 「未設定」「空」「false」を区別する必要があるか、それを差分が表現できているか
  - `null` の意味（不明 / 空 / N/A）が業務上一貫しているか

### B-3. 時刻・期間境界
- **境界の包含**: 期間指定（「`from` 以上 `to` 未満」/「`from` から `to` まで」）が
  業務仕様・既存 API・UI 表記と一致しているか。
- **タイムゾーン境界**: ユーザーから見た「今日」と内部 UTC の境界がずれて、日次集計や有効期限が
  ずれていないか（午前 0 時前後のテストが必要なケース）。
- **DST / 夏時間**: 該当地域の業務で、存在しない時刻 / 二重に存在する時刻が結果に影響しないか。
- **期間の長さ**: 0 秒 / 0 日 / 同日内 / 跨日 / 跨月 / 跨年 / 跨閏年 / 跨閏秒 が要件範囲に入るかを確認。

### B-4. 並行・順序境界
- **同時操作**: 同じリソースに対する 2 並行操作（同じ商品を同時にカートに入れる、
  同じクーポンを同時に適用するなど）の挙動が要求通りか。要求が定めていない場合、
  暫定挙動が安全側か（ロックや冪等性は `robust-safety` D / `reliability` で扱うが、
  **要求として定義されているべき**ケースはここで指摘）。
- **順序入れ替わり**: イベントが想定順と違って到着した場合の業務上の正しい結果が
  定義されているか。

### B-5. ロケール・国際化
- 言語・地域差で結果が変わり得る経路（数値書式・通貨・住所・氏名順序・電話番号フォーマット・
  言語別の単複形 等）で、要件と実装が一致しているか。
- ソート / 比較がロケール依存（Turkish I 問題、ドイツ語 ß など）で結果が違う経路を含んでいないか。
  業務要件として locale-sensitive なソートが必要なら、その指定が明示されているか。

### B-6. 権限・テナント境界
- 同じ操作でも**ロール / 権限 / テナント**で結果が変わるべきフロー（「管理者なら全件、
  一般ユーザーなら自分のだけ」）で、すべての分岐が要求通りに動くか。
- 認可そのものは `security` カテゴリだが、「要求として定められた `挙動の差` が実装に
  反映されているか」は本カテゴリで指摘する。

## C. 契約（API / 型 / スキーマ / イベント）との整合

外部に対する**約束**（コントラクト）と、実装の挙動が一致しているか。
コントラクトの **ソース・オブ・トゥルース**（OpenAPI / GraphQL schema / proto / TypeScript 型 /
JSON Schema 等）が定義されている場合、それと差分の整合を最優先で確認する
（プロジェクト方針: コントラクト層を厳密に定義し、実装層は再生成可能に保つ）。

### C-1. API コントラクト
- **OpenAPI / GraphQL / gRPC proto** などのコントラクト定義が存在する場合、差分が定義に従っているか:
  - 必須フィールドの有無
  - 型・enum 値・nullability
  - エンドポイント URL / メソッド / クエリパラメータ
  - レスポンスステータスコードと意味の一致
- コントラクトと実装が**ドリフト**していないか。手書きの型・モデルが先行して、
  コントラクト定義の更新が忘れられていないか。
- コントラクトを **本変更で書き換えた** 場合、その変更が破壊的でないか、
  破壊的であれば C-4（後方互換性）で扱う。

### C-2. 型・スキーマ
- 関数シグネチャ・公開型が、呼び出し側の期待と整合するか。
  TypeScript で `any` / `unknown` / `as` キャストで型穴を作っていないか
  （静的検査可能な部分は型チェッカに委ね、ここでは **意味的なズレ** を見る:
  `"id"` フィールドの型が変わって既存呼び出しが壊れる、null 許容の幅が広がっている、など）。
- DB スキーマの変更が、アプリ層のモデル定義と一致しているか
  （マイグレーション安全性は `database` D 節で扱う）。

### C-3. ビヘイビア仕様（仕様書 / Acceptance Criteria）
- 仕様書 / 設計ドキュメントに記述された**振る舞い**（順序・タイミング・出力形式・通知タイミング 等）が、
  差分の挙動と一致しているか。
- 仕様書がない場合、PR 説明や Issue から読み取れる「合意された振る舞い」と一致しているか。

### C-4. 後方互換性（仕様レベル）
- **既存クライアント**を壊す変更になっていないか:
  - フィールド削除 / リネーム / optional → required
  - 既存 enum 値の意味変更 / 削除
  - エラーコード・ステータスコードの意味変更
  - イベントスキーマの破壊的変更（旧 consumer が新メッセージで落ちる、逆も然り）
- 破壊的変更となる場合:
  - **deprecation の経路**（並行運用期間 / `Deprecation` ヘッダ等）が用意されているか
  - 段階的ロールアウト（追加 → 両対応 → 切替 → 旧削除）に分解できているか（`_reviewer-stance.md` B 節と連動）

### C-5. イベント・メッセージ・ジョブキュー
- **イベントスキーマ**を生成 / 消費する経路で、旧フォーマットのメッセージが inflight にある状態で
  新コードが落ちないか / 新フォーマットを旧コードが受け取っても落ちないか。
- ジョブキュー（Sidekiq / Cloud Tasks / SQS 等）に**未処理ジョブ**があるとき、新ハンドラが旧引数形式を
  受けても安全か。逆も然り。
- メッセージの **at-least-once 配信**前提で、消費側が冪等になっているか（仕様要件として）。

## D. エラーパスと結果の表現

「どう失敗するか」と「失敗をどう伝えるか」を**仕様視点**で見る。
コード上の例外設計・fail-safe の網羅性は `robust-safety` H 節で扱い、
失敗時のメッセージ文言・UX は `usability` で扱う。本節はその間の **「業務上、どんな失敗を区別して、
何を返すと約束したか」** を見る。

### D-1. エラーケースの定義と表現
- 仕様で定義された**エラーケース** が、実装側で**区別して**返されているか。
  すべて 500 / generic error にまとめていないか。
- ステータスコード / エラーコード / 例外型 が、**呼び出し側がどう動けば良いかを判別できる粒度**か:
  - リトライ可能か不可能か（一時的失敗 vs 恒久的失敗）
  - ユーザー入力を直すべきか、システム側の問題か
  - 認証 / 認可の失敗の区別（ただし enumeration リスクは `security` F-3）
- エラーレスポンスが**機械可読な識別子**（error code）を含んでいるか。文字列メッセージだけに依存して
  クライアントが分岐していないか。

### D-2. 失敗時の戻り値・型
- `Result<T, E>` / `Either` / 例外 / 戻り値 + bool / null返却 等、**プロジェクトのエラー伝播方式**と
  整合しているか（同 PR 内で混在していないか）。
- 「エラー時に空配列を返す」「エラー時に null を返す」のような暗黙の規約が、呼び出し側と一致しているか。
  「データなし」と「失敗」が混同されていないか。

### D-3. 部分成功・部分失敗
- 一括処理（バッチ作成 / 一括更新）で、**部分成功**を返す必要があるか / 全件失敗で扱うか の選択が
  要求と一致しているか。
- 部分成功を返す場合、各要素の成否が呼び出し側で判別可能な構造か。
- 失敗件数 / 成功件数 / 失敗理由が要求された粒度で取れるか。

### D-4. ロールバック・補正
- 失敗時に**やったことを取り消す**必要がある経路（外部通知済み・課金済み・ファイル作成済み）で、
  仕様として補正フローが定義されているか / 実装に反映されているか。
- 取り消し不能な副作用（送信済みメール）を**前提とした注意喚起 / 再送防止**が仕様に含まれているか。

## E. 状態・条件の組合せ網羅

機能が複数の条件 / 状態 / フラグで分岐する場合、**組合せが網羅的にカバー**されているか。

### E-1. 真理値表 / 状態遷移網羅
- 複数の bool / enum で分岐するロジックで、**起こり得るすべての組合せ**に対する期待挙動が
  仕様 / テスト / コードに表現されているか。
- 状態遷移を持つオブジェクト（注文・予約・ジョブ）で、各遷移の前提・結果が定義されているか。
  「`pending` → `cancelled` → `paid`」のような不正遷移を防いでいるか
  （型レベルの遮断は `robust-safety` B-3、本節は仕様としての遷移定義の有無を見る）。

### E-2. 排他的状態の整合
- 「同時に true になり得ない」前提のフラグが、実装上で本当に排他になっているか。
  仕様上は `Idle | Loading | Success | Error` のはずが、実装で `isLoading` + `isError` の bool で
  `Loading かつ Error` になり得る状態が生まれていないか
  （型レベルの議論は `robust-safety` B-2）。

### E-3. 複合フィルタ・複合検索
- フィルタ条件が複数あるとき、**AND / OR の合成**が要求通りか。
  「カテゴリ A かつ ステータス active」で「A の active のみ」を返すべきか「A 全部 + active 全部」
  を返すか、設計が明確か。
- 検索結果のソート順が、複合条件下で安定（stable）か。同点時のタイブレークが定義されているか。

### E-4. プロパティ間の整合
- 「`status === 'paid'` のとき `paid_at` が必須」のような**プロパティ間の依存**が、
  生成時 / 更新時の各経路で破られていないか
  （型での結合は `robust-safety` B-3、本節は仕様としての依存関係の定義 / 実装反映を見る）。

## F. 副作用・冪等性（仕様視点）

副作用（DB 書き込み・外部 API 呼び出し・通知送信・ファイル変更）が、**業務要件として**
正しい範囲・回数・順序で発生するかを見る。
メッセージング基盤の at-least-once 配信に対する冪等化は `reliability` カテゴリだが、
**仕様として「2 回呼んでも 1 回扱い」と定めているか**は本カテゴリの責務。

### F-1. 同じ操作の繰り返し
- 同じ操作を複数回呼んだとき、要求された挙動になるか:
  - 冪等であるべきか（決済確定 / 招待送信 等で重複させない）
  - 累積であるべきか（ポイント加算 等で増やしていく）
  - その方針が仕様で明示されているか / コード上で担保されているか

### F-2. 副作用の対象・範囲
- 副作用の**対象**が要求通りか（誰に通知が飛ぶか / どのテナントの DB が書き換わるか）。
- 副作用の**順序**: 「DB 更新 → 通知 → 外部 API」の順序が仕様要件として正しいか。
  失敗時にどこまで実行されたかが、業務上問題ない範囲か。

### F-3. 観測可能な副作用と内部副作用
- 業務上意味のある副作用（メール送信・課金・在庫減算）と、内部キャッシュ更新等の副作用が
  分離して扱われているか。
- 監査ログ / イベント発行が、**業務イベントとして必要な粒度**で出ているか
  （ロギング詳細は `reliability` / `security` J 節、本節は「業務上必要なイベントが出ているか」のみ見る）。

## G. テスト・受け入れ検証

### G-1. 新挙動に対するテストの存在
- 差分が**新しい挙動・分岐・エッジケース**を導入している場合、それに対するテストが追加されているか。
- 既存テストの修正だけで済まされている場合、その理由（既存テストが新挙動も網羅している / 上位の
  E2E でカバーされている）が説明できるか。
- プロジェクト方針が **TDD（探索 → Red → Green → Refactoring）** の場合、
  テストが**先行作成**された痕跡（コミット履歴）が確認できるか
  （CLAUDE.md: TDD で開発する）。

### G-2. テストが「契約」を検証しているか
- テストが**実装内部の手順**ではなく、**外部から見える契約**（入力 → 出力 / 状態遷移 / 副作用）を
  検証しているか。リファクタで壊れるテスト（モック呼び出し回数を厳密に検査するなど）に偏っていないか
  （`maintainability` 3 節と連動）。

### G-3. エッジケースカバレッジ
- B 節 / E 節で挙げた境界・組合せのうち、**業務上重要なもの**にテストが付いているか。
- 不正入力 / エラーパス / 失敗時挙動に対するテストがあるか
  （ハッピーパスだけのテスト構成になっていないか）。

### G-4. 再現条件・固定種
- ランダム性 / 時刻依存 / 外部 I/O に依存するテストで、**固定 seed / 固定時刻 / モック / fixture**が
  使われているか。フレーキー（不安定）になりやすい構造になっていないか。
- 並行テスト・統合テストで、テスト間のデータ干渉が起きないか
  （並列実行可能性、teardown の徹底）。

### G-5. 手動検証 / Smoke
- UI / フロントエンドの差分で、**ブラウザでの実体験**による検証が PR 説明・チェックリストで
  確認できるか。型チェック・自動テストだけでは UI 機能の正しさは保証されない（CLAUDE.md 方針）。
- リスクの高い差分（マイグレーション / 認可変更 / 課金処理）で、ステージング検証や手動シナリオが
  PR チェックリストに含まれているか。

## H. データの意味的正しさ

### H-1. 単位・スケール・形式
- 数値が**単位**（円 / セント / ms / s / バイト / KB / 日 / 時間）と一致しているか。
  関数を跨ぐ際に単位変換が正しく行われているか（変換漏れで 1000 倍ずれる事故が典型）。
- 日時が **UTC** で保持されているか、表示時に正しく変換されているか（`robust-safety` F-3 と連動するが、
  本節は **業務として正しい単位・タイムゾーンで返しているか** を見る）。
- 表示用フォーマット（カンマ区切り・パーセント・通貨記号）と内部値が混同されていないか。

### H-2. 集計の包含基準
- 集計（合計・平均・件数）の **対象範囲**が要求と一致しているか:
  - 論理削除済みデータを含めるか
  - キャンセル済み / 失敗済みのレコードを含めるか
  - 期間境界（開区間 / 閉区間）が一致しているか
  - 重複排除の基準が要求通りか
- 「税抜 / 税込」「割引前 / 割引後」「為替変換前 / 後」の選択が、**表示と内部値**で一貫しているか。

### H-3. データの存在・欠損の意味
- 「データがない」状態が、業務上**どの意味**を持つか:
  - まだ取得していない（未取得） vs 該当なし（empty）
  - 0 件 vs null vs undefined vs エラー
  これらが API レスポンス / UI 表示で要求通りに区別されているか。

### H-4. 並び順の安定性
- 一覧表示・検索結果の並び順が、**仕様で定義された順序**で安定しているか。
  同点時のタイブレーク（id 昇順 / 作成日時降順 等）が定義されていて、ページング時にも崩れないか。

## 共通: severity 判定の目安（本カテゴリ固有の補足）

`_reviewer-stance.md` A-3 を基本に、本カテゴリでは以下を加味する:

| 区分 | severity の目安 |
|------|----------------|
| 受け入れ基準が明確に未達成、または既存機能を壊している | blocker |
| エッジケースの一部で要求どおりに動かない / 契約と実装がドリフト / 主要パスのテスト欠落 | major |
| 仕様で定義のないが想定外の挙動になる入力経路 / 周辺の改善余地 | minor |
| 命名・テスト名・好みの範囲 | nit |

- **「明確に動かない」よりも、「要求と実装の乖離が見える」**ほうが本カテゴリの本領。
  攻撃者視点（security）や保守視点（maintainability）と違い、**ユーザーが意図した結果が返るか**で判定する。
- 受け入れ基準が文書化されていないために判断不能な指摘は、severity を minor に下げて
  「要求の明確化が必要」として残す（要求欠落そのものを blocker / major にしない）。

## 除外・スコープ外

- **コードレベルの不正状態の不可表現化・並行/例外設計** — `robust-safety` カテゴリ
  （業務要件として漏れているケースは本カテゴリで扱う）
- **エラーメッセージ文言・空状態コピー・操作 UX** — `usability` カテゴリ
- **失敗時のリトライ戦略・冪等性の実装メカニズム・観測性** — `reliability` カテゴリ
  （仕様として「2 回呼んでも 1 回扱い」と定めているか自体は本カテゴリで扱う）
- **計算量・スループット・レスポンスタイム** — `performance` カテゴリ
- **コードの読みやすさ・命名・凝集度** — `maintainability` カテゴリ
- **認可ロジックそのものの脆弱性・攻撃者視点の評価** — `security` カテゴリ
  （「要求として定められた振る舞いが実装されているか」は本カテゴリ）
- **DB スキーマ・マイグレーションの安全性** — `database` カテゴリ
- **API 設計の HTTP セマンティクス（メソッド・URI 形）** — `backend` A 節
  （定められた API 仕様と実装が一致しているかは本カテゴリ）
- **静的検査で検出可能なルール**（型エラー・lint 違反 等）— 各プロジェクトの linter / 型チェッカ

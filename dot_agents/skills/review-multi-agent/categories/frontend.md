# frontend 観点

## このカテゴリで見るもの
ブラウザ / モバイル / デスクトップ UI のコード（React / Vue / Svelte / React Native 等）、および UI から見た
API 呼び出しパターン・状態管理・UI 特有の型設計を対象とする。
「レビュワーの設計判断が必要な観点」だけをここに列挙する。フォーマット・lint ルール・型エラー等の
機械的チェックは各プロジェクトの linter / formatter / tsc に委ねる前提（CLAUDE.md 方針）。

本ファイルの記述は React を主軸にしているが、考え方は他フレームワークでも応用可能。
プロジェクト固有のコンポーネント分類規約（例: `action / model / component`）や
ディレクトリ構成は `AGENTS.md` / `docs/frontend/` を優先する。

## A. コンポーネント設計

UI コードの保守性は「責務の切り方」と「副作用の置き場所」でほぼ決まる。
ここでは reviewer の設計判断が必要な観点のみを扱う。

### A-1. 責務の単一性
- 1 コンポーネントが「**表示**」「**ローカル状態保持**」「**副作用（mutation / 通信 / 外部 I/O）**」を
  同時に抱えていないか。役割が複数あるコンポーネントは、ボタン単位 / アイテム単位に分解できないか。
- プロジェクト規約で `action / model / component` 等の層分けが定義されている場合、
  コードがその層境界を越えて結合していないか。
- 「画面コンポーネント」が、データ取得・整形・分岐・表示・サブミット を全部 1 ファイルで抱え込んでいないか。

### A-2. 粒度の妥当性
- 過剰結合: メニュー + 確認モーダル + 削除実行 を 1 コンポーネントに詰めていないか。
  各責務（メニュー開閉 / 削除確認 / 削除実行）は別コンポーネントに切れないか。
- 過剰分割: 「1 画面で 20 個の小コンポーネントをただ並べるだけ」になっていないか。
  分割の理由（再利用 / テスト境界 / 再レンダー局所化）が説明できるか。

### A-3. 状態・副作用の配置
- イベントハンドラ・`useMutation` 等のカスタムフック呼び出しを、可能な限り**葉コンポーネント**に寄せているか。
  親コンポーネントに集約することで、無関係な兄弟コンポーネントまで再レンダーに巻き込んでいないか。
- 「削除ボタンの責務（モーダル表示 / キャンセル / 実行）」と「リストアイテムの責務」が混在していないか。

### A-4. Props 設計
- Props 命名が「データ」と「コールバック」と「フラグ」で区別されているか
  （`onXxx` / `isXxx` / `hasXxx` などプロジェクト規約に従う）。
- `children` を渡せば済むところで、抽象的な props slot（`header` / `footer` / `content` 等）を
  乱立させていないか。逆に、複数の挿入箇所が必要な場合に `children` 1 本で無理していないか。
- Optional な props のデフォルト値が、呼び出し側ではなくコンポーネント側に集約されているか
  （同じデフォルトが各呼び出し元に散らばっていないか）。

### A-5. Container / Presentational の線引き
- データ取得・状態管理を担う層（Container / hook）と、純粋表示を担う層（Presentational）の境界が明確か。
- Presentational コンポーネントに `useQuery` / `useMutation` / グローバルストア参照が紛れ込んでいないか
  （Storybook 単体・テスト単体での検証性を損なう）。
- 「全部のコンポーネントを Container 化」「全部 Presentational」のような極端な振り方になっていないか
  （プロジェクトの一般方針と整合しているか）。

### A-6. 共通化のタイミング
- 2 箇所の類似コードはまだ抽象化せず残し、3 箇所目以降で共通化する原則から外れて、
  「1 箇所からしか呼ばれない共通コンポーネント / カスタムフック」を先に作っていないか。
- 逆に、4 箇所以上で重複している UI / ロジックを放置していないか。
- 共通化された結果、呼び出し側が大量の prop を渡してカスタマイズしている＝**早すぎる抽象化**になっていないか。

## B. React レンダリング / フック

React 固有の「動かないわけではないが、後で必ず痛む」パターンを抑える節。
リンタで検出できるもの（exhaustive-deps の単純違反など）は除外し、設計判断が必要な観点のみ扱う。

### B-1. `useEffect` の必要性
- その副作用は本当に「**レンダー起因**」か。イベント起因の副作用（ボタンクリックでリクエスト送信、
  フォーム送信時のトラッキング 等）を `useEffect` + 依存配列で擬似的に扱っていないか。
- state とハンドラの再配置（state を親に上げる / 派生値は計算で出す / イベントハンドラに直接書く）で、
  effect そのものを消せないか検討されているか。
- 「props が変わったら state を更新する」「props から state を初期化してそのまま使う」など、
  React 公式「You Might Not Need an Effect」で名指しされているアンチパターンに該当していないか。

### B-2. `useEffect` のクリーンアップ
- 購読 / タイマー / イベントリスナ / 外部 DOM 参照 / `AbortController` に対するクリーンアップが書かれているか。
- StrictMode の二重マウント・依存配列変化時の再実行で、リソースリーク・重複購読・古いリクエストの
  レース状態が起きない設計になっているか（マウント中に発火した古い fetch を破棄しているか）。

### B-3. 値のキャプチャ起因のバグ
- 古い props / state をクロージャに閉じ込めたまま、後から呼ばれるコールバック / setTimeout / 購読ハンドラで
  参照していないか（依存配列に載せ忘れ、または `useRef` / `useEffectEvent` 相当での回避が必要なケース）。
- イベントハンドラ内で「最新の state を取りたい」場面に対し、適切な手段（関数型 setState / ref ミラー）が
  選ばれているか。

### B-4. 再レンダリングのトリガと範囲
- 親の state 更新で巻き込まれる子の範囲が意識されているか。コストの高いツリー（リスト・グラフ・エディタ）が
  無関係な親の state 変化で再レンダーされていないか。
- `Context.Provider` の `value` が毎回新オブジェクトになっていないか（`useMemo` / 安定参照で渡されているか）。
- Context を使っているが「変化頻度の高い値」と「変化頻度の低い値」を 1 つの Context に詰めて、
  全消費者を毎回再レンダーさせていないか（Context を分割する選択肢が検討されているか）。

### B-5. 最適化（memo / useCallback / useMemo）の妥当性
- `memo` / `useCallback` / `useMemo` を「再レンダーを減らす」目的で**とりあえず**付けていないか。
  計測・プロファイル・体感問題の根拠なしに導入していないか
  （Dan Abramov「Before You memo()」: コンポジション変更で済まないかを先に検討）。
- 逆に、明らかに毎レンダー走ると重い計算（大きな配列のフィルタ・ソート・集計）を素で書いていないか。
- `useCallback` で渡したハンドラの参照安定が、子の `memo` と組み合わさって**実際に効いている**か
  （子側が `memo` でなければ意味がない）。

### B-6. state の保持位置（リフトアップ / リセット）
- 同じ位置で描画されるコンポーネントが、ID 切り替え時に `key` で意図的にリセットされているか
  （フォームの初期値が前のレコードのまま残る、等のバグを防げているか）。
- 逆に、リセットしたくない state（入力中のテキスト 等）が、親の `key` 変更や条件分岐の片側に置かれて
  予期せずリセットされていないか。
- state を**上に持ち上げる**判断と**下に下ろす**判断（コロケーション）の選択が、共有範囲に対して妥当か。

### B-7. カスタムフック化
- 2 つ以上のフック呼び出しで構成されるロジックが、凝集性のあるカスタムフックに切り出されているか。
- 既存フックを**ラップして使い方を縛る**意図のカスタムフック（例: `useQuery` を内部隠蔽して、
  必要な引数 / 戻り値だけ露出する）が必要な場面で素の React フックを撒いていないか。
- カスタムフックが「ただの関数で済む処理（フックを呼ばない）」を `use` プレフィックスで包んでいないか。

### B-8. 避難ハッチ（ref / 直接 DOM 操作）
- スクロール位置制御 / フォーカス管理 / サードパーティライブラリ統合 / 計測 など、
  **宣言的表現では届かない部分だけ**に ref / 直接 DOM 操作が限定されているか。
- 宣言的に書ける箇所（条件付き表示・クラス切り替え・スタイル）を ref で書いていないか。
- 自作コンポーネントに ref を渡す場合、`forwardRef`（または React 19 以降の ref-as-prop）の流儀が
  プロジェクト方針と一致しているか。

## C. 状態管理

「どの状態をどこに置くか」と「サーバ状態と UI 状態を混ぜないか」が中心。
ライブラリ選定そのものは個別案件、ここではレビュー時の判断基準を示す。

### C-1. state の所在判断
- その状態は次のどこが正か、を意識して配置されているか:
  1. コンポーネント内 `useState`（1 コンポーネント内で完結）
  2. 親コンポーネントへリフトアップ（兄弟間で共有）
  3. Context（深くネストされた読み取り専用 / 低頻度更新の値）
  4. グローバルストア（複数ルートで共有 / 永続化される UI 状態）
  5. サーバキャッシュ（API レスポンス由来）
- ローカルで済むものをグローバルに置いていないか。
  逆に 3 画面で共有されるものをコンポーネント内に閉じて prop バケツリレーしていないか。

### C-2. サーバ状態と UI 状態の分離
- API レスポンスを `useState` に格納し、`useEffect` で再フェッチ・キャッシュ無効化を**手書き**していないか。
  React Query (TanStack Query) / SWR / RTK Query 等の導入が検討されているか
  （プロジェクトに既存の選定がある場合はそれに従う）。
- サーバキャッシュ層を導入済みのプロジェクトで、**今回の差分だけ**手書きの fetch + state に逆戻りしていないか。

### C-3. `useQuery` 系フックの責務
- サーバキャッシュから読む（あるいは派生する）だけで済むデータを、カスタムフック内で
  `useState` にコピーしていないか（同期バグの温床 / オリジナルとの乖離リスク）。
- `useQuery` をラップしたカスタムフックが、UI 都合のローカル state（ダイアログ開閉 等）を
  紛れ込ませていないか（責務混在）。

### C-4. 複雑な状態更新ロジック
- 3 つ以上の関連 state を「同時に整合性を保ちつつ」更新する箇所で、
  `useReducer` への集約が検討されているか。分散した `setState` で順序依存・中間矛盾が起きていないか。
- 「直前の state を見て次の state を決める」処理で、関数型 setState（`setX(prev => ...)`）が使われているか。

### C-5. ライブラリ選定
- Zustand / Redux / Jotai / Recoil / Context のどれを使うか、**プロジェクトの既存スタックと整合**しているか。
- 既存に Redux があるのに今回だけ Zustand を入れる、などの場合、必要性の説明が PR 説明 / 設計ドキュメント
  にあるか。逆に既存の選定理由が古びていて見直す価値があるかは別議論。
- 「Context だけで足りるのに Redux を入れる」「Redux 大規模プロジェクトに Zustand を散発的に追加する」など、
  チーム全体の認知負荷を増やす方向に動いていないか。

## 除外・スコープ外
- **a11y の原理・ガイドライン全般 / WCAG 準拠の網羅評価** → `usability` カテゴリ
  （本ファイル H 節は実装に密着した部分のみ）
- **XSS / CSRF / `dangerouslySetInnerHTML` の安全性 / 認証トークンの取り扱い** → `security` カテゴリ
- **bundle size / コード分割 / Lighthouse・Profiler によるレンダリング性能計測** → `performance` カテゴリ
- **テスト構造の良し悪し・テスト観点そのもの** → `maintainability` / `functionality` カテゴリ
- **要求仕様の充足・エッジケース挙動** → `functionality` カテゴリ
- **エラーメッセージ・UI コピー・空状態の文言** → `usability` カテゴリ
- **失敗時のリトライ戦略・冪等性** → `reliability` カテゴリ
- **CI / ビルドツール（Vite / Webpack / SWC / Babel）の設定そのもの** → `infrastructure` カテゴリ
- **ESLint / Prettier / tsc で機械的に拾える項目すべて**（コードフォーマット・命名規則・未使用変数・
  exhaustive-deps の単純違反 等）

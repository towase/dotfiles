# performance 観点（優先順位: 7）

## このカテゴリで見るもの
**「想定するワークロードに対して、必要な計算量・リソース効率・応答速度・スループットが
得られているか」** を見る。具体的には:

- アルゴリズム計算量（Big-O）
- データベース・永続化アクセスパターン（N+1 / インデックス利用）
- メモリ使用と GC 圧
- ネットワーク・外部 I/O の効率
- 並行・並列化と非同期の設計
- キャッシュ戦略と無効化
- フロントエンド性能（Core Web Vitals）
- 性能回帰の検出と観測性

主な参照: **Donald Knuth "premature optimization is the root of all evil"**、Brendan Gregg
*Systems Performance*、Martin Kleppmann *Designing Data-Intensive Applications*、
Google **web.dev** / **Core Web Vitals**、各 RDBMS の `EXPLAIN`/`ANALYZE` ドキュメント。

機械的に検出できる範囲（lint で検出可能な明白な無駄、bundler の analyze、ベンチランナーの結果）は
その環境のツールに委ね、ここでは「**人間の見立てが必要な性能トレードオフ**」を扱う。

## 観点構成
- A. 計測前提と最適化方針（measure first / hot path 特定）
- B. 計算量とアルゴリズム
- C. データベース・永続化アクセスパターン
- D. メモリと割当
- E. ネットワーク・外部 I/O
- F. 並行・並列化と非同期
- G. キャッシュ戦略
- H. フロントエンド性能（Core Web Vitals）
- I. 観測性と性能回帰検出

## A. 計測前提と最適化方針

性能観点で**最も避けたい指摘**は「**計測なしの最適化提案**」。Knuth の言葉通り、根拠なき最適化は
複雑性を増やすだけで失敗する確率が高い。本カテゴリでは「**計測 / 見立てが妥当か**」「**ホットパスで
明らかに非効率な構造になっていないか**」を中心に判断し、推測ベースの最適化提案には慎重になる。

### A-1. 性能要件 / SLO の有無
- レビュー対象が**明示的な性能要件**（応答時間 / スループット / メモリ上限 / Core Web Vitals 目標 等）に
  紐づくか。要件があれば**それに照らして**判定する。要件不明なら「**現状から悪化していないか**」を
  最低ラインにする。
- `reliability` H-3 の SLO / エラーバジェット運用がある場合、性能変更がバジェットにどう影響するかが
  意識されているか。

### A-2. ホットパスとコールドパス
- 差分が**ホットパス**（高頻度実行 / クリティカルパス / レイテンシ予算が厳しい経路）に該当するか:
  - リクエストのメインパス
  - リスト描画 / 表示画面の初期化
  - ループ内 / バッチの内側
  - リアルタイム処理（音声 / 動画 / ゲーム）
- ホットパスの最適化提案は厳しめに見る（成果が大きい）。コールドパスでの最適化提案は
  保守性を損なうトレードオフを慎重に。

### A-3. 計測の有無
- 性能を変える差分が、**ベンチマーク / プロファイル**に裏付けられているか。
  「速くなったはず」「重そう」だけの判断で複雑性が上がっていないか。
- 既存ベンチマーク・性能テストへの影響が確認されているか。

### A-4. 提案の粒度
- `_reviewer-stance.md` A 節の通り、性能改善提案も**最小コスト案を本命**にする。
  「全体アーキテクチャ書き換え」を前面に出す前に、**ホットスポット 1 箇所の修正**で十分かを判定する。
- マイクロ最適化（ループ展開・命令選択）の提案は、**測定された差**がない限り原則 minor / nit に下げる。

## B. 計算量とアルゴリズム

### B-1. 入力サイズの見立て
- 関数 / クエリ / 処理に渡る**入力サイズ N の最大想定**が分かるか。N が小さい（<100）なら O(N²) でも
  問題ない場合があり、N が大きい（>10⁵）なら O(N log N) でも厳しい場合がある。
- **N の境界が将来増える**経路（ユーザー数 / 商品数 / 履歴件数）で、現在の素朴実装が将来詰まらないか。

### B-2. 不要な O(N²) / O(N³)
- ループの**中**で `array.includes` / `array.indexOf` / `Object.keys().find` のような線形検索が
  あり、外側ループと組み合わさって O(N²) / O(N×M) になっていないか。
- 集合演算（含有判定 / 重複排除 / 差集合）に `Array` を使い続けて、`Set` / `Map` / `HashMap` への
  置き換えが見えていないか。

### B-3. ソート・探索
- ソート不要な処理にソートを差し込んでいないか（一覧表示用に最大値を取りたいだけなら**1 パス**）。
- 既にソート済みのデータを**再ソート**していないか。
- 二分探索が使える局面で線形探索のままになっていないか。

### B-4. 早期終了・lazy 評価
- 結果が確定した時点で**早期 return / break** できる箇所で、最後まで走り切る実装になっていないか。
- ストリーム処理（generator / iterator）で済ませられる箇所を、**全件実体化**してからフィルタ・
  マップしていないか（メモリにも影響）。

### B-5. キャッシュ可能な計算
- 同じ入力で同じ結果になる**純粋な重い計算**が、ループの内側で繰り返されていないか
  （ループ外への巻き上げ / メモ化）。
- 派生値の計算（合計・統計）が、毎回**全件再計算**になっていないか。差分更新で済むか。

### B-6. データ構造の選択
- 主要操作の頻度（読み中心 / 書き中心 / 検索中心）と、選んだデータ構造（配列 / 連結リスト / ハッシュ /
  ツリー / トライ）が整合しているか。
- 用途に対して**過剰な構造**（B-Tree が必要ない場面で B-Tree、ヒープが必要ない場面で priority queue）が
  入っていないか。

### B-7. アルゴリズムの正しさ
- 性能改善のために変えたアルゴリズム / クエリが、**結果として元の出力と一致**するか
  （`functionality` E / G と連動）。並び順・タイブレーク・null 扱いがズレていないか。

## C. データベース・永続化アクセスパターン

`database` C 節（インデックス戦略）が**スキーマ側**を見るのに対し、本節は**コードからのアクセス
パターン**を見る。

### C-1. N+1 クエリ
- ループの中で 1 件ずつクエリを投げるパターンが残っていないか。
  ORM の lazy 関連参照（Active Record / Hibernate / Sequelize / Prisma 等）が**意図せず発火**して
  いないか。
- 解消方法（`includes` / `JOIN` / `preload` / `DataLoader` / `eager` / バッチローダ）が、
  そのフレームワーク慣習に沿って選ばれているか。
- **サーバ側 N+1**だけでなく、GraphQL / BFF からの**クライアント側 N+1**（フィールド単位で取得が
  発生する）も同様に見る。

### C-2. SELECT 全件・SELECT *
- `WHERE` 条件 / `LIMIT` 無しの全件取得を**メモリに乗せて**から後段でフィルタしていないか。
  DB 側でフィルタすべき。
- 必要なカラムだけ `SELECT col1, col2` で絞れる場面で、`SELECT *` で巨大なテキスト・JSON カラムを
  毎回引いていないか。

### C-3. インデックス利用の確認
- 主要なクエリが**期待するインデックス**を使うように書かれているか
  （関数適用で sargable でなくなる / `LIKE '%xxx'` の前方ワイルドカード / 暗黙の型変換）。
- 複合インデックスの**左端一致**を破る形でクエリが書かれていないか
  （詳細は `database` C-1）。

### C-4. ページング・ストリーミング
- 大量データを返す可能性がある経路に**ページング**が掛かっているか
  （`backend` A-5 / `database` H と連動）。
- 大規模 export / 集計でメモリに全件積まずに **cursor / streaming**（`COPY` / `UNLOAD` / SAX 系）で
  処理できるか。
- ページングが**カーソル / キーセット**で実装されるべき大規模一覧で、`OFFSET` のままになっていないか
  （末尾ページが O(N)）。

### C-5. 書き込みの効率
- 大量 INSERT / UPDATE が**1 件ずつ**ではなく**bulk** で発行されているか。
- DB ラウンドトリップを伴う処理が**トランザクション境界の中**で過剰に膨らんでいないか
  （`reliability` C-1 / `database` E-1 と連動）。

### C-6. 集計・分析クエリ
- OLTP テーブルに対する重い集計（フルスキャン / 大規模 GROUP BY）が、本番のオンライン経路で
  発火していないか。マテリアライズドビュー / 集計テーブル / 別 DWH への切り出しが検討されているか
  （`database` H 連動）。

### C-7. 接続プール
- 接続数 / プールサイズが、想定同時接続と DB 側上限に整合しているか。
- 長時間トランザクション・長時間クエリで**プールを食い潰す**経路がないか。

## D. メモリと割当

### D-1. 大きなデータの一括読み込み
- 行数が多くなり得るデータを**全件メモリに展開**していないか。
- 大きな JSON / XML / CSV を全文文字列として保持していないか。streaming パーサ（SAX 系 /
  `JSONStream` / `csv-parse stream` / `Encoding.NewDecoder` 等）に置き換える価値があるか。

### D-2. ホットパスでの割当
- 高頻度に呼ばれる関数 / ループ内で**毎回オブジェクト生成・配列生成・文字列連結**が走っていないか
  （GC 圧 / アロケーション）。
- バッファ / 配列の**再利用**（`Buffer.allocUnsafe` の使い分け / `sync.Pool` 等）が
  必要なホットパスで使われているか。

### D-3. 不要なコピー
- `[...arr]` / `Object.assign({}, x)` / `slice()` のような**コピーが連鎖**して、入力サイズに対して
  M 倍のメモリを使っていないか。
- 不変性のために必要な防御的コピー（`robust-safety` I-2）と、**惰性コピー**を区別する。

### D-4. メモリリーク傾向
- イベントリスナ / サブスクリプション / タイマー / コールバックが**解除されない**経路がないか
  （`robust-safety` E-3 と連動）。
- 長寿命オブジェクトが短寿命オブジェクトを**捕捉**して GC を阻害していないか。
- グローバル / シングルトンへの**追加だけ**で削除がない経路がないか（cache without eviction）。

### D-5. 文字列とバイト
- 大きな文字列をバイトに変換 / 逆変換を**何度も**繰り返していないか。
- マルチバイト境界での処理（UTF-8 のサロゲート / 結合文字）が、不必要に**全件 copy** を引き起こしていないか。

## E. ネットワーク・外部 I/O

### E-1. ラウンドトリップ削減
- 外部 API への呼び出しがループ内で 1 件ずつ走っていないか。
  バッチエンドポイント / 並列化 / バルク取得の選択肢が検討されているか。
- DB / キャッシュ / 外部 API への問い合わせを、**1 リクエスト内で何回**しているかが見える形か。

### E-2. ペイロードサイズ
- レスポンスボディが必要以上に膨らんでいないか（不要フィールド / 巨大 base64 / 全文返却）。
  `backend` A-5 のフィールド絞りが効いているか。
- 圧縮（gzip / brotli）が効くべき経路で効いているか。
- `Cache-Control` / `ETag` / `If-None-Match` で**条件付き取得**が成立しているか（G 節と連動）。

### E-3. 並列化と直列化
- 互いに依存しない外部呼び出しが**直列**になっていないか（`Promise.all` / `errgroup` / 並行アクター
  での並列化）。
- 逆に、依存先のレートリミットを破る並列化になっていないか（`reliability` A-5）。

### E-4. プロトコル選択
- HTTP/1.1 vs HTTP/2 / HTTP/3、REST vs GraphQL vs gRPC、WebSocket vs SSE vs polling の
  選択が、用途と一致しているか（特にラウンドトリップ多 / 双方向 / ストリーミングが要件のとき）。

### E-5. CDN / エッジ活用
- 静的 / 半静的なコンテンツが**CDN / エッジキャッシュ**で配信されているか
  （詳細は `infrastructure` カテゴリだが、コードからのキャッシュ可能性ヒント設定 — `Cache-Control` /
  `Vary` / `Surrogate-Key` 等 — は本カテゴリ）。

## F. 並行・並列化と非同期

### F-1. CPU バウンド vs I/O バウンド
- 処理の性質（CPU バウンド / I/O バウンド）が見立てられ、**選んだ並行モデル**（イベントループ /
  スレッド / プロセス / WebWorker / goroutine）と整合しているか。
- I/O バウンドの async/await を**直列に await** で繋いで並列化機会を捨てていないか
  （E-3 と連動）。
- CPU バウンドな重い処理を**メインスレッド / イベントループ**でブロックしていないか
  （Node.js: `worker_threads`、ブラウザ: Web Worker）。

### F-2. 並行度の制御
- 並列実行の数に**上限**が設定されているか（`p-limit` / semaphore / worker pool）。
  上限なしの `Promise.all` で外部依存を爆撃していないか。
- 並行度が低すぎてリソースを使い切れていない（過小並列）も同時に見る。

### F-3. 非同期 API の正しい利用
- `await` 抜けで Promise が裸で投げられていないか（`robust-safety` D-4 と連動するが、性能視点では
  期待した順序での実行が崩れていないか）。
- 非同期処理のキャンセル / タイムアウトが効くか（`reliability` A-1 / `robust-safety` D-3 と連動）。

### F-4. 共有可変状態とコンテンション
- 並列実行で**ロック競合**が頻発する設計になっていないか。
  hot row / hot key への集中アクセスを分散できないか（カウンタ分割 / シャーディング / batch flush）。

## G. キャッシュ戦略

キャッシュは**正しさ**と**性能**の両方を同時に扱うため、性能観点での提案でも**整合性**が壊れないか
を必ず併せて見る（`functionality` C-2 / `security` E-3 / `reliability` C と連動）。

### G-1. 階層
- どこに置くか:
  - クライアント（HTTP cache / IndexedDB / SWR / React Query）
  - CDN / エッジ
  - リバースプロキシ / API Gateway
  - アプリ内（in-process / shared in-memory）
  - 共有キャッシュ（Redis / Memcached）
- 階層の選択が、**鮮度要件 / 共有範囲 / コスト**と整合しているか。

### G-2. 鍵設計
- キャッシュキーが**ユーザー / テナント / ロケール / バージョン**を含むべき場合に含まれているか
  （他人のデータが見える事故 = `security` 領域に直結）。
- キーが過剰に細かく分かれてヒット率が出ない / 過剰に粗くて衝突する、のどちらにも倒れていないか。

### G-3. 無効化（invalidation）
- TTL / イベント / 手動のいずれかが選ばれ、**書き込み時の整合性**が設計されているか。
  「書いたけど読み戻したらキャッシュが返ってきて古いまま」が想定外のフローで起きていないか。
- キャッシュ無効化のコスト（fan-out 量 / Redis 通信）が運用上問題にならないか。

### G-4. キャッシュスタンピード / dogpile
- TTL 切れと同時に**多数のリクエストがバックエンドへ殺到**する経路に、対策（lock / 早期再生成 /
  jitter / probabilistic early expiration）があるか。

### G-5. キャッシュ汚染とポイズニング
- ユーザー入力に基づくキー / コンテンツが、**他ユーザーの応答に混入**する経路がないか
  （Cache Poisoning は `security` E にも該当）。
- `Vary` ヘッダ / cache key normalize の不備で、別ユーザー応答が混入しないか。

### G-6. ネガティブキャッシュ
- 失敗 / 404 / 空結果のキャッシュが必要なら入れているか / 不要なら入れていないか。
  「DB 落ちて全部 500 が返って、それを各層がキャッシュして全画面 500」のような事故を防げるか。

## H. フロントエンド性能（Core Web Vitals）

ブラウザ / モバイル UI 側の差分は、ユーザー体感性能に直結する。
**Core Web Vitals**（Google が定義する 3 指標）が現状の代表的目標:

| 指標 | 内容 | "Good" 閾値 |
|------|------|------------|
| **LCP** (Largest Contentful Paint) | 最大コンテンツ描画までの時間 | ≤ 2.5s |
| **INP** (Interaction to Next Paint) | UI 反応性（2024-03 に FID を置き換えた現行指標） | ≤ 200ms |
| **CLS** (Cumulative Layout Shift) | レイアウトシフト累積 | ≤ 0.1 |

### H-1. バンドルサイズ
- 新しい依存追加で**bundle が大きく増えていない**か。代替（軽量ライブラリ / 部分インポート /
  自前実装）が比較されたか。
- **tree shaking** が効くインポート（named import / ES module）になっているか。
  `import * as X` で全部取り込んでいないか。
- **コード分割**（route-based / component-based / dynamic import）で、初期表示に不要なコードを
  遅延ロードできているか。

### H-2. LCP（初期描画）
- ヒーロー画像 / メインテキストの**ブロッキングリソース**を減らせるか
  （preload / preconnect / fetchpriority="high"）。
- フォント読み込みが LCP をブロックしていないか（`font-display: swap` / 自前ホスト）。
- サーバ側レンダリング / 静的生成（SSR / SSG / ISR）で First Paint を早められる経路か。

### H-3. INP（インタラクション）
- 入力イベントハンドラ内で**メインスレッドを長くブロック**する処理が無いか
  （重い計算 / 同期的な大規模 DOM 操作）。
- 必要なら `requestIdleCallback` / `scheduler.postTask` / Web Worker で**外に出す**選択肢が検討されたか。
- 不要な再レンダー（`frontend` B-4 連動）が INP に影響していないか。

### H-4. CLS（レイアウトシフト）
- 画像 / 広告 / iframe / 動的挿入要素に**サイズ予約**（`width` / `height` / `aspect-ratio` /
  プレースホルダ）があるか。
- フォント切り替え時の**FOIT / FOUT** で大きな字形シフトが起きていないか
  （`size-adjust` / `font-size-adjust`）。

### H-5. 画像・メディア最適化
- 適切な**フォーマット**（AVIF / WebP / JPEG XL / SVG）が選ばれているか。
- 適切な**サイズ**（responsive / `srcset` / `sizes` / 適切な解像度）か。
- **遅延読み込み**（`loading="lazy"` / Intersection Observer）が効くべき箇所で効いているか。

### H-6. ネットワーク Waterfall
- **クリティカルレンダリングパス**にあるリソース数が最小か。
- **preload / prefetch / preconnect / dns-prefetch** が、効果のある対象に絞って使われているか
  （誤用は逆効果になる）。

### H-7. Hydration / SSR
- SSR / SSG / RSC の hydration が、初期インタラクション可能になるまでの時間を膨らませていないか
  （島アーキテクチャ / 部分 hydration / `client:visible` 系の戦略）。

### H-8. データ取得
- リスト・無限スクロールで `frontend` B / C と連動した最適化（仮想スクロール / 部分更新 /
  pagination / SWR キャッシュ）が効いているか。

## I. 観測性と性能回帰検出

性能は**測れない経路で気づけない**。本節は性能の可観測性と CI での回帰検出を見る。

### I-1. 性能メトリクスの可観測性
- 主要エンドポイント / 重要処理の**レイテンシ分布**（p50 / p95 / p99）が観測できているか
  （`reliability` G-2 RED と連動）。
- DB クエリ・外部 API・キャッシュヒット率などの**レイヤ別計測**が分散トレースで取れるか
  （`reliability` G-3 連動）。
- フロントエンドは **Real User Monitoring (RUM)** / Web Vitals 計測が入っているか。

### I-2. ベンチマーク
- 性能要件を持つ箇所に**ベンチマーク**（micro-bench / load test / Lighthouse）があるか。
- 本変更がベンチマークの**結果を更新**しているか / 既存ベンチマークを壊していないか。

### I-3. 性能回帰の自動検出
- CI で**bundle size 監視** / **Lighthouse CI** / **load test 自動実行**が動いているか。
- 回帰閾値（例: `+10%` で警告）が設定され、PR コメントに反映される運用か。

### I-4. プロファイリングのしやすさ
- 新規コードに対して**プロファイラがフラットな結果を返せる**形か（過度な抽象化で「全部 anonymous
  関数」にならないか）。
- `pprof` / Chrome DevTools / `perf` などで観察したときに、ボトルネックが追えるか。

## 共通: severity 判定の目安（本カテゴリ固有の補足）

`_reviewer-stance.md` A-3 を基本に、本カテゴリでは以下を加味する:

| 区分 | severity の目安 |
|------|----------------|
| 本番想定の N で**明らかに動かない / 落ちる**（OOM・タイムアウト・DoS 類似）/ 本番計測で SLO 大幅違反が確実 | blocker |
| 本番想定の N でレイテンシが顕著に劣化 / N+1 などの典型悪パターンがホットパスにある / Core Web Vitals が "Poor" 帯に落ちる | major |
| 局所的な改善余地、現状の N では問題ないが将来悪化し得る | minor |
| 推測ベース / 計測なしの提案、定数倍のマイクロ最適化 | nit |

- **「本番に近い N と頻度」を仮置きしてシナリオを描けないなら**、severity は控えめに。
  `_reviewer-stance.md` A 節の通り、推測ベースの提案は最小コスト案・別 finding として残す。
- **計測 / プロファイル**を伴う提案は重く扱う。「速くなりそう」だけの提案は minor / nit に下げる。
- 性能のために**正しさ・可読性・運用容易性**を犠牲にする提案は、トレードオフを明記する。

## 除外・スコープ外

- **DB スキーマ・インデックス設計そのもの** — `database` C / H 節
  （本カテゴリはコードからのアクセスパターン）
- **計算量爆発を悪用した DoS（ReDoS / zip bomb / 巨大 payload 等の意図的攻撃）** — `security` I-2 / C-5
  （本カテゴリは正常負荷下での性能のみ）
- **可用性 / 復旧 / 障害時挙動** — `reliability` カテゴリ
- **コードの読みやすさ・抽象化のトレードオフ** — `maintainability` カテゴリ
  （ただし**性能のために可読性を捨てる提案**は本カテゴリで「トレードオフ明記」して扱う）
- **計算結果の正しさ自体** — `functionality` カテゴリ
- **インフラ層のスケール / オートスケール / k8s リソース設定** — `infrastructure` カテゴリ
- **lint / 自動 bundle analysis で検出可能なルール** — 各 CI

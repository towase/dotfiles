# database 観点

## このカテゴリで見るもの
DB スキーマ・制約・インデックス・マイグレーション・トランザクションなど、永続化レイヤの設計判断を対象とする。
backend カテゴリの「データアクセス層のコード」とは切り口を分け、こちらは **データそのものの構造・運用・進化** を見る。
機械的に検出できるルール（SQL 構文・命名規約等）は linter / migration ツールに委ね、ここでは
「**設計判断と運用影響**」だけをレビュー対象とする。

## A. スキーマ設計

### A-1. テーブル粒度・集約境界
- テーブルが業務上の **集約境界** と一致しているか。1 テーブルに複数の独立したライフサイクルが同居していないか。
- 1:1 で必ず付随する属性を別テーブルに切る設計の根拠が明確か（NULL 列削減 / 機密分離 / 物理 I/O 分離など）。
- 共通化を狙った汎用テーブル（EAV、`type` + `payload JSON`）が、本来別テーブルに分けるべき概念を潰していないか。

### A-2. 主キー戦略
- **代理キー（surrogate）か自然キー（natural）か** の選択根拠が一貫しているか。プロジェクト内で混在する場合は理由が説明できるか。
- ID 型の選択（`bigint` / `uuid` / `ulid` / `bigserial`）が、生成主体（DB 生成 / アプリ生成 / クライアント生成）と一貫しているか。
- 公開 ID（外部に晒す ID）と内部 PK を分離すべきケース（推測可能性・enumeration 攻撃）が考慮されているか。

### A-3. 列の型・サイズ
- 文字列長制限・数値精度・日時型（`timestamp` / `timestamptz` / `date` / `time`）の選択が業務要件と一致しているか。
- 金額を `float` / `double` で持っていないか（精度要件があれば `decimal/numeric` か整数で最小単位）。
- 真偽値を `tinyint(1)` 等で曖昧に表現していないか。`bool`/`boolean` を素直に使えるか。
- enum を **DB 側 ENUM 型** にするか、**lookup table + FK** にするか、**アプリ側のみで管理** するかの選択根拠が明確か（変更頻度・他システム連携の有無で決まる）。

### A-4. NULL 許容方針
- `NULL` が「未入力」「不明」「該当なし」のどれを意味するかが列ごとに整理されているか。
- 「未入力 vs 空文字 vs NULL」のように複数表現を許していないか。
- 業務上必須の列に対して、追加時のデフォルト値設計と既存行のバックフィルが計画されているか。

### A-5. 多態的関連
- ポリモーフィック関連（`target_type` + `target_id`）を導入する場合、**FK 制約が効かない** トレードオフを把握した上で採用しているか。
- 代替手段（per-type 中間テーブル / 抽象テーブル + 具象テーブルの継承）が比較検討されているか。

### A-6. 命名・規約の一貫性
- 列名・テーブル名のケース・複数形/単数形・接頭辞がプロジェクト規約と一致しているか
  （規約自体は linter/CI で見るが、**規約に存在しないパターンが新規で出現** していないかは人間の判断）。
- `created_at` / `updated_at` / `deleted_at` などのタイムスタンプ列が、用途・タイムゾーン方針（UTC 保存）に従っているか。

## B. 制約 / 不変条件

### B-1. NOT NULL
- 業務上必須の列に **NOT NULL** が付いているか。アプリ層だけで担保していないか。
- 「将来必須化したいが移行中」の列は、その旨が明示されているか（コメント / マイグレーション計画）。

### B-2. UNIQUE
- 業務上の自然キー（メールアドレス、外部システムの ID など）に UNIQUE 制約があるか。
- 論理削除と UNIQUE が衝突する場合（同じメアドの再登録など）、**部分インデックス** や `deleted_at` を含む複合 UNIQUE で解消されているか。
- 複合 UNIQUE の **列順** が意味的に正しいか（参照クエリ・FK 用途と整合）。

### B-3. 外部キー（FK）
- 参照整合性が必要な関連に FK が張られているか。アプリ層でのみチェックして DB に張らない選択は、その理由（マルチテナント分離 / 非同期書き込み / シャーディング）が明確か。
- `ON DELETE` / `ON UPDATE` の挙動（`CASCADE` / `RESTRICT` / `SET NULL` / `NO ACTION`）が業務意図と一致しているか。
- 大規模テーブルへの FK 追加時のロック影響が考慮されているか（D-1 と連動）。

### B-4. CHECK 制約・式制約
- DB 側で表現できる不変条件（範囲・列間関係・状態遷移）が CHECK で強制されているか、それともアプリ層に委ねる方針か。判断基準が一貫しているか。

### B-5. アプリ層 vs DB 層の責務分担
- 「複数経路（バッチ・管理画面・API）から書き込まれる可能性」がある不変条件は **DB 側** に寄せられているか。
- アプリ層のみで担保する場合、その経路が単一であることが保証できているか。

## C. インデックス戦略

### C-1. クエリパターンとの整合
- 主要なクエリ（`WHERE` / `ORDER BY` / `JOIN` / `GROUP BY`）に対して、適切なインデックスが用意されているか。
- 複合インデックスの **列順** が、クエリの選択性とアクセスパターンに沿っているか（左端一致原則）。
- カバリングインデックス（INCLUDE / 含み列）にする価値があるホットパスかが評価されているか。

### C-2. 過不足
- 同じ列セットに対して重複・冗長なインデックスがないか（複合の左端と単独の重複など）。
- 書き込み頻度の高いテーブルにインデックスを追加する際、書き込みコスト増が見合うか。
- 低カーディナリティ列単独のインデックスが効果薄になっていないか（部分インデックス / 複合化を検討したか）。

### C-3. 特殊インデックス
- 部分インデックス（`WHERE`）・式インデックス・GIN/GiST・全文検索インデックスの選択が、クエリと一致しているか。
- 大量データの `LIKE 'foo%'` / 前方一致・後方一致・全文検索の使い分けが妥当か。

### C-4. 命名・運用
- インデックス名が一意かつ役割が分かるか（自動生成名のままで運用上困らないか）。
- 不要になった旧インデックスの削除が、マイグレーションに含まれているか。

## D. マイグレーション安全性

### D-1. オンライン DDL とロック影響
- 対象 RDBMS で **テーブルロックを伴う DDL** が走っていないか（特に大規模テーブルへの `ALTER`）。
- ロック影響がある場合、**メンテナンスウィンドウ** か **オンライン DDL ツール**（pt-online-schema-change / gh-ost / `ALTER TABLE ... ALGORITHM=INPLACE` 等）の利用方針が決まっているか。

### D-2. ロールバック可否
- マイグレーションが **ロールバック可能** な形式（`down`/`reverse` 定義）になっているか、もしくは forward-only 方針で運用上の合意があるか。
- データ削除を伴うマイグレーションは、復旧不能であることが明示され、別ステップに分離されているか。

### D-3. expand / contract パターン
- 破壊的変更（列の削除・リネーム・型変更・NOT NULL 化）が **段階分け** されているか:
  1. 新カラム追加（expand）
  2. 両書き / バックフィル
  3. 旧カラム参照削除
  4. 旧カラム削除（contract）
- 各段階の間にデプロイサイクルが入る前提で、**旧コードと新スキーマ・新コードと旧スキーマ** の双方向で動くか。

### D-4. バックフィルの戦略
- 大規模テーブルの一括 UPDATE を 1 トランザクションで流していないか（ロック・レプリ遅延・バイナリログ肥大）。
- バッチ化・スループット制御・進捗観測（再開可能性）が組み込まれているか。
- バックフィル中の新規書き込みとの整合性（書き込み時にも新カラムを埋める）が考慮されているか。

### D-5. 失敗時の中断可能性
- マイグレーションが途中で失敗した場合の状態が把握できるか（部分適用が残らないか、残るなら検出・修復手順があるか）。
- DDL と DML を 1 トランザクションに混ぜる前提が、対象 DB の動作（PostgreSQL は可、MySQL は暗黙コミット）と整合しているか。

### D-6. 環境差
- ステージング・本番で **データ規模** や **レプリ構成** が異なる場合、検証環境での所要時間が本番見積もりに使えるかが意識されているか。

## E. トランザクション・整合性

### E-1. トランザクション境界
- 1 つの業務不変条件を守るための更新が、**同一トランザクション** に収まっているか。複数トランザクションに分割している場合、その間の中間状態が外から観測可能でも問題ないか。
- 逆に、トランザクションが過大（外部 API 呼び出し・長時間処理を含む）になっていないか。

### E-2. 分離レベル
- 既定の分離レベル（READ COMMITTED / REPEATABLE READ など）と、本変更が要求する分離レベルが整合しているか。
- ロストアップデート・ファントムリード・ノンリピータブルリードのいずれかが業務上問題になる経路で、適切な対策（行ロック `FOR UPDATE` / 楽観ロック / SERIALIZABLE）が取られているか。

### E-3. ロック・並行性
- 複数行・複数テーブルにまたがる更新の **ロック取得順序** が統一され、デッドロックを誘発しにくい設計になっているか。
- ホットスポット（特定行の連続更新）になる設計を避ける手段（カウンタ分割・キュー化）が検討されているか。

### E-4. 分散トランザクション回避
- DB を跨ぐ整合性が必要に見える場面で、**2PC を避ける** 代替（Saga / Outbox / 結果整合性 + 補償処理）が検討されているか。
- 採用された方式の **失敗ケース** と再実行時の冪等性が説明できるか。

## F. 履歴・監査・論理削除

### F-1. 履歴保持
- 業務要件として「過去のある時点の状態」を再現する必要があるか、その実現方法（履歴テーブル / イベントソーシング / temporal table / バージョニング列）が要件に合っているか。
- 履歴粒度（行単位 / 列単位 / 操作単位）と保持期間が明確か。

### F-2. 論理削除（soft delete）
- `deleted_at` 等を導入する場合、**全クエリで除外する** 仕組み（共通スコープ / view）が用意されているか。除外漏れによるデータ復活リスクが評価されているか。
- FK や UNIQUE 制約が論理削除と整合するか（B-2, B-3 と連動）。
- 物理削除との使い分け方針（PII 削除権利 / アーカイブ要件）が明確か。

### F-3. 監査ログ
- 「誰が・いつ・何を変えたか」が必要な対象に、監査ログの仕組みがあるか（DB トリガ / アプリ層イベント / CDC）。
- 監査ログ自体の改竄防止・保持期間が要件と整合しているか。

## G. 容量・運用・分割

### G-1. 増加見込み
- 想定データ増加量に対して、現在のスキーマ・インデックスが数年スケールで成立するか。
- パーティショニング（レンジ / リスト / ハッシュ）の必要性と、パーティションキーの選択が業務クエリと整合するか。

### G-2. アーカイブ
- 古いデータの分離戦略（コールドストレージ / 別テーブル / TTL）が、要件と運用コストの間で妥当か。

### G-3. マルチテナンシー
- テナント分離方式（DB 分離 / スキーマ分離 / `tenant_id` 列）が、隔離要件・運用コスト・コスト効率のバランスで選ばれているか。
- `tenant_id` 列方式の場合、**全クエリ・全 UNIQUE 制約** に `tenant_id` が含まれているか。

### G-4. 文字コード・タイムゾーン・コラレーション
- 文字コード（`utf8mb4` 等）・コラレーション・タイムゾーン（UTC 保存）の方針がプロジェクトで統一され、本変更がそれに従っているか。

### G-5. バックアップ・リストア
- 復旧時に「失われ得る最大の窓」（RPO）が明確で、バックアップ方式と整合しているか。
- スキーマ変更後にバックアップからリストアする場合の互換性が確認されているか。

## H. データアクセスパターンとの整合

- 想定する主要クエリが、現在のスキーマ + インデックスで効率的に表現できるか。
- 集計・レポーティング系の重いクエリが OLTP テーブルに直接走る設計になっていないか（マテリアライズドビュー / 集計テーブル / DWH への切り出しを検討したか）。
- 「アプリ層の N+1」自体は `performance` カテゴリ扱いだが、**スキーマがそれを誘発している**（例: 必要な情報を別テーブルから取らないと組み立てられない構造）場合は本カテゴリで指摘する。

---

## I. 機密データ / PII

- PII / 機微情報が **どの列** に存在するかが特定可能か（カラムコメント / メタデータ）。
- 暗号化（at rest / カラムレベル / アプリ側暗号化）方針が要件と一致しているか。
- 削除権利（GDPR の right to be forgotten 等）に対応するための物理削除パスが設計されているか。
- ログ・バックアップ・レプリカ・分析基盤への PII 流出経路が把握されているか（詳細は `security` カテゴリで深掘りする）。

---

## 除外・スコープ外
- データアクセス層のコード品質・ORM の使い方（`backend` カテゴリ）
- 認証・認可・暗号鍵管理の脆弱性そのもの（`security` カテゴリ）
- アプリ層起因の N+1・実行計画チューニングの細部（`performance` カテゴリ）
- DB サーバ/レプリカの構成・バックアップジョブの実装（`infrastructure` カテゴリ）
- 失敗時の冪等性・整合性の全体像（`reliability` カテゴリ）

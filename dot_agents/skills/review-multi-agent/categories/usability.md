# usability 観点（優先順位: 5）

## このカテゴリで見るもの
**「使う側にとってこの差分は使いやすいか / 使えなくなっていないか」** を見る。
ユーザーは 1 種類ではない:

- **エンドユーザー**（Web / モバイル / デスクトップ UI を直接触る人）
- **API クライアント開発者**（HTTP / GraphQL / gRPC / SDK / CLI を呼ぶ人）
- **運用者・オペレーター**（CLI / 設定ファイル / ログ / アラートを介してシステムを動かす人）
- **読者**（README / ドキュメント / リリースノートを読む人）

主な参照: **Nielsen の 10 Usability Heuristics**、**WCAG 2.2**（2023-10、ISO/IEC 40500:2025 として 2025-10 認定）、
W3C **ARIA Authoring Practices Guide (APG)**、Microsoft *Inclusive Design*、Don Norman *The Design of
Everyday Things*。

機械的に検出できる範囲（HTML lint・axe-core / Lighthouse のヒューリスティクス・色コントラスト計測）は
その環境のツールに委ね、ここでは「**実体験 / 認知 / 文脈の判断が必要な観点**」を扱う。
**プロジェクト規約**: エラーメッセージは可能な限り「**次のユーザーのアクション**」を伝える
（CLAUDE.md 方針）。

## 観点構成
- A. ユーザータイプとタスク完遂
- B. 状態の提示（loading / empty / error / success）
- C. エラーメッセージと次のアクション
- D. フォーム・入力 UX
- E. ナビゲーション・情報設計
- F. アクセシビリティ（WCAG 2.2 / POUR）
- G. 国際化・ローカリゼーション
- H. API のユーザビリティ（DX / Developer Experience）
- I. 運用者向けユーザビリティ（CLI / 設定 / ログ / ドキュメント）

## A. ユーザータイプとタスク完遂

差分を見る前に「**この変更で誰が何をしたくなる/しにくくなるか**」を特定する。
実体験を伴わないレビューは表面的になりやすいので、**想定ユーザーで一度操作する**目線を持つ。

### A-1. ユーザータイプの特定
- 影響を受けるユーザー像（エンドユーザー / API クライアント / 運用者 / 内部ツール利用者）が、
  PR 説明や該当機能のコンテキストから読み取れているか。
- **複数のユーザータイプ**にまたがる差分（管理画面 + 公開 API 等）で、片方への配慮が薄くなっていないか。
- アクセシビリティ要配慮ユーザー（視覚 / 聴覚 / 運動 / 認知）が含まれるユースケースで、
  該当の配慮が組み込まれているか（F 節）。

### A-2. ターゲットタスクの完遂可能性
- 想定タスクが**詰まることなく完了できる**か（Nielsen #7: Flexibility and efficiency of use）。
  代表的なフローを頭の中でなぞって、**詰まる箇所**（次に何を押すか分からない / 画面遷移後に必要情報が無い 等）が無いか。
- **マウスのみ / キーボードのみ / 画面読み上げのみ / モバイルタッチのみ**のいずれでも基本タスクが達成できるか。
- 「達成できるが**やたら手数が多い**」になっていないか（既存フローの劣化）。

### A-3. ユーザーコントロールと取り消し（Nielsen #3）
- 重要な操作（削除 / 送信 / 公開 / 変更）に対して、**取り消し / 確認 / 戻る**経路があるか。
- 確認ダイアログが**重要な変更にだけ**出ているか（多用するとアラートピロー化して効果を失う）。
- 部分的な入力途中で離脱したときに**入力内容を失う**経路がないか（D-5 と連動）。

## B. 状態の提示（loading / empty / error / success）

Nielsen #1: **Visibility of system status**。

### B-1. 4 状態の網羅
新しい画面 / 新しい操作で、以下の 4 状態すべてが**意図的に設計**されているか:
- **Loading**: 処理中のフィードバックがあるか（スピナー / スケルトン / プログレス）。
  長時間（>3 秒）処理は進捗 / キャンセル可能か。
- **Empty**: データが無いとき、空白ではなく**手引き**（「まだ X がありません。Y から作成できます」）が
  出るか。初回利用者に対するオンボーディングを兼ねているか。
- **Error**: 失敗時に何が起きたか + 何ができるかが提示されるか（C 節）。
- **Success**: 完了後のフィードバック（トースト / インライン表示 / 画面遷移）があるか。
  「ボタンを押したのに何も起きないように見える」状態を作っていないか。

### B-2. 待機中の手応え（Perceived Performance）
- ボタンクリック直後の**即時フィードバック**（disabled 化 / ローディング表示）があるか。
  二重送信を防止しつつ、「クリックされた」が即座に伝わるか（D-4 と連動）。
- 楽観的更新（optimistic UI）を使う場合、**失敗時の戻し**が破綻していないか。

### B-3. 進捗の粒度
- 長い処理（アップロード / 一括処理 / レポート生成）で、**進捗率 / 残り時間 / 件数**のいずれかが
  示せるなら示しているか。
- 進捗が見せられない処理でも、**何かが動いている**ことが感じられる工夫があるか。

## C. エラーメッセージと次のアクション

**プロジェクト規約**: エラーメッセージはなるべく「**次のユーザーのアクション**」を伝える（CLAUDE.md）。
Nielsen #9 (Help users recognize, diagnose, and recover from errors) と直結する最重要観点の 1 つ。

### C-1. 「何が起きたか」+「何ができるか」
- エラーメッセージが**現象の説明だけ**で終わっていないか:
  - ❌ 「エラーが発生しました」「Bad Request」「500 Internal Server Error」
  - ⭕ 「メールアドレスの形式が正しくありません。`@` を含めてください」
  - ⭕ 「サーバが混み合っています。1 分後に再試行してください」
- ユーザーが**できないこと**ではなく**できること**で書かれているか。
- 「サポートに問い合わせてください」と書く場合、問い合わせ先 / 参照すべき ID が併記されているか。

### C-2. エラーの位置づけ
- エラーが**発生箇所の近く**に出ているか（フォームならその入力フィールドの直下）。
  ページ全体トーストだけで、どのフィールドが問題か分からない経路になっていないか。
- 複数エラーがあるとき、**個別に表示**されるか（最初の 1 件で打ち止めになって、他のエラーが見えないと修正のたび再送信が必要）。

### C-3. 機械可読 + 人間可読の二層
- API レスポンスのエラーで、機械可読な **error code**（クライアント側で分岐できる識別子）と、
  人間可読な **message** が両方含まれているか（`backend` A-5 と連動）。
- 文字列メッセージだけに依存してクライアントが switch していないか。

### C-4. 内部実装の漏洩抑制（security 連動）
- スタックトレース / SQL / 内部パス / ライブラリ名 がエラーメッセージに漏れていないか
  （`security` F-3 と連動）。本カテゴリでは「**ユーザーにとって意味のない技術詳細が表に出ていないか**」
  という UX 観点でも見る。

### C-5. 国際化対応
- エラーメッセージが**ハードコードされた英語 / 日本語**ではなく、翻訳可能な構造になっているか
  （G 節と連動）。ユーザー向けコピーが**ソースコード文字列リテラル**として散在していないか。

### C-6. 「ユーザーが原因ではない」エラーの扱い
- サーバ障害・依存先障害など**ユーザー責任ではない**エラーで、ユーザーを責める文言（「不正なリクエスト」
  「無効な操作」など）になっていないか。

## D. フォーム・入力 UX

### D-1. 入力支援
- **placeholder と label の使い分け**: placeholder は label の代替にしていないか（フォーカス後に消える）。
  入力例は help text として別途示されているか。
- 入力フォーマットの**例示**（電話番号 `090-1234-5678` 等）が、対応する地域 / 言語に合致しているか。
- **オートコンプリート**（`autocomplete="email"` 等）が、適切な input 種別に付与されているか。

### D-2. バリデーションのタイミング（Nielsen #5: Error prevention）
- 検証が**いつ実行**されるか（onChange / onBlur / onSubmit）が、ユーザーの入力速度・誤入力許容度と
  整合しているか。タイプ中の真っ赤な「無効」表示で、まだ入力途中なのに責められていないか。
- **必須項目の未入力 / 形式違反 / 重複**などの違いが、メッセージで区別されているか。

### D-3. 必須・任意・制限の明示
- 必須項目が**事前**に分かるか（送信時に初めて気付かされていないか）。
- 文字数 / ファイルサイズ / 選択数の上限が、入力前に示されているか（カウンター・上限表示）。
- 受け付ける形式（拡張子 / MIME）が事前に示されているか。

### D-4. 二重送信・送信中の状態
- 送信ボタンが**送信中に disabled**になるか（D-1 / B-2 と連動）。
- バックエンド側の冪等化（`reliability` B-2）と組み合わせて、**ネットワーク二重送信**でも安全か。

### D-5. 入力の保護
- ページ離脱・誤操作で**入力内容が失われない**か（confirm dialog / 自動下書き保存）。
- 戻るボタンで前画面の入力が**保持されている**か（react の `key` リセット問題は `frontend` B-6）。

### D-6. アクセシブル認証（WCAG 3.3.8 AA）
- ログイン / 認証フローで、**記憶 / 認知タスクへの依存**だけが残っていないか。
  パスワードマネージャ自動入力 / 生体認証 / マジックリンク等の代替が許容されているか。

## E. ナビゲーション・情報設計

### E-1. 現在地の手がかり（Nielsen #1）
- ユーザーが**どこにいるか**が分かるか（パンくず / アクティブメニュー / タイトル / URL）。
- 深いページから**戻る経路**が破綻していないか（リロード後 / 直接 URL 開封後でも戻れるか）。

### E-2. 一貫性と標準（Nielsen #4）
- 既存画面と**用語 / 並び順 / アイコン / 配置**が一貫しているか。今回の差分だけが例外パターンに
  なっていないか。
- プラットフォーム慣習（モバイル: 下タブ / デスクトップ: 左サイドバー など）と整合しているか。

### E-3. 情報の階層と段階的開示
- 重要情報がスクロールせず見えるか。重要度の低い情報が**段階的に開示**されているか
  （詳細リンク / アコーディオン / モーダル）。
- 大量情報を 1 画面に詰め込みすぎて**重要なものが埋もれていない**か（Nielsen #8）。

### E-4. Recognition over Recall（Nielsen #6）
- ユーザーが**覚えていないと進めない**経路を作っていないか
  （前画面で見た値を手で入力させない / 候補から選ばせる）。

### E-5. 一貫したヘルプ（WCAG 3.2.6 AA）
- 複数ページで提供される**ヘルプ**（連絡先・FAQ・チャット）が、**同じ相対順序の場所**にあるか。
  画面ごとに位置が変わると、認知障害のあるユーザーが見つけにくい。

## F. アクセシビリティ（WCAG 2.2 / POUR 原則）

WCAG 2.2 は **POUR** 4 原則で構成: **Perceivable / Operable / Understandable / Robust**。
**Level AA** 適合を最低ラインとして見る（多くの法令・調達基準の参照点）。
**実装に密着した a11y**（`button` vs `div+onClick` / `htmlFor` / `alt` / `aria-label` の付与漏れ）は
`frontend` H 節で扱う。**本節は WCAG 原則の充足度・配慮の方向性**を見る。

### F-1. Perceivable（知覚可能）
- **代替テキスト**（WCAG 1.1.1 A）: 意味のある画像に `alt` があり、装飾画像は `alt=""` で
  読み上げから除外されているか。動画にキャプション / トランスクリプトがあるか。
- **コントラスト**（WCAG 1.4.3 AA / 1.4.11 AA）: テキスト 4.5:1（大きい文字 3:1）、
  UI コンポーネント / 状態表現 3:1 が確保されているか。**色のみで情報を伝えていない**か（1.4.1 A）。
- **拡大耐性**（WCAG 1.4.4 AA / 1.4.10 AA）: テキスト 200% 拡大 / 320 CSS px 幅でのリフロー破綻が無いか。
- **動きとアニメーション**（WCAG 2.3.3 AAA / `prefers-reduced-motion`）:
  パララックス / 自動再生 / 過剰アニメで認知負荷を上げていないか。OS 設定の reduced motion を尊重しているか。
- **ライブリージョン**: 状態変化（送信完了 / エラー発生）が**画面読み上げに伝わる**か（`aria-live`）。

### F-2. Operable（操作可能）
- **キーボード操作**（WCAG 2.1.1 A / 2.1.2 A）: マウスなしで全機能に到達 / 操作 / 離脱できるか。
  フォーカスの罠（trap）に陥らないか。
- **フォーカス可視性**（WCAG 2.4.7 AA / 2.4.11 AA / 2.4.13 AA）:
  - キーボードフォーカスが**常に視認**できるか
  - 固定ヘッダ / モーダルでフォーカスが**遮られていない**か（2.4.11 AA, 新規）
  - フォーカスインジケータが十分なコントラスト・面積を持つか（2.4.13 AA, 新規）
- **タップターゲットサイズ**（WCAG 2.5.8 AA, 新規）: 最低 24×24 CSS px。
- **ドラッグ操作の代替**（WCAG 2.5.7 AA, 新規）: ドラッグでしかできない操作に**シングルポイントの代替**
  （ボタン / メニュー）があるか。
- **タイムリミット**（WCAG 2.2.1 A）: 自動セッション切れ / 自動進行に**延長 / 解除 / 警告**があるか。

### F-3. Understandable（理解可能）
- **言語指定**（WCAG 3.1.1 A）: ページ / 部分の `lang` 属性が正しいか（読み上げ言語が正しく切り替わる）。
- **予測可能性**（WCAG 3.2.x）: フォーカスやフォーム入力で**予期しない画面遷移**を起こさないか。
- **エラー特定と修正案**（WCAG 3.3.1 A / 3.3.3 AA）: C 節と直結。
- **冗長入力の回避**（WCAG 3.3.7 AA, 新規）: 同じセッション内で**既に入力した情報を再入力**させていないか
  （オートフィル / 持ち越し）。
- **アクセシブル認証**（WCAG 3.3.8 AA, 新規）: 記憶やパズルだけに頼る認証になっていないか（D-6）。

### F-4. Robust（堅牢）
- **セマンティック HTML / ARIA**: ネイティブ要素（`button` / `a` / `nav` / `main` / `form`）優先。
  ARIA は**ネイティブで表現できないとき**にだけ使う（"No ARIA is better than Bad ARIA"）。
  詳細実装は `frontend` H 節。
- **支援技術の互換性**: スクリーンリーダー（VoiceOver / NVDA / TalkBack）でテストされる前提があるか、
  少なくとも構造的に支援技術が読める形になっているか。

### F-5. WCAG 2.2 新規追加（AA）の押さえ
WCAG 2.2 で AA に追加された主な基準を、押さえどころとして再掲:
- 2.4.11 Focus Not Obscured (Minimum)
- 2.4.13 Focus Appearance (Minimum)
- 2.5.7 Dragging Movements
- 2.5.8 Target Size (Minimum)
- 3.2.6 Consistent Help
- 3.3.7 Redundant Entry
- 3.3.8 Accessible Authentication (Minimum)

## G. 国際化・ローカリゼーション

### G-1. 翻訳容易性
- ユーザー向け文字列が**ハードコード**されていないか。リソースファイル / i18n キーを介しているか。
- メッセージに**変数を埋め込む**箇所で、語順が言語によって変わる前提（ICU MessageFormat 等）に
  なっているか。文字列連結で `"Hello, " + name` のような構築になっていないか。

### G-2. 単複・性別・敬語
- 「1 件」「N 件」を `plural` ルールで処理しているか。英語 / 日本語以外の複雑な複数形（ロシア語 / アラビア語）
  に対応する余地があるか（必要な場合）。
- 性別 / 敬語による語形変化が必要な言語（ドイツ語 / フランス語 / 韓国語）への配慮が必要なら設計されているか。

### G-3. 数値・日付・通貨フォーマット
- 数値（小数点 `.` vs `,` / 桁区切り）・日付（`YYYY-MM-DD` vs `MM/DD/YYYY`）・通貨記号 / 表記順が
  ロケールに応じて切り替わるか（`Intl.NumberFormat` / `Intl.DateTimeFormat` 等）。
- 内部表現は UTC / ISO 8601 / ISO 4217 / 整数最小単位など**規格準拠**で保持され、
  表示時にだけロケール変換しているか（`robust-safety` F / `functionality` H と連動）。

### G-4. RTL（右から左言語）
- アラビア語 / ヘブライ語などの RTL 言語サポートが必要な場合、レイアウト（border / margin / padding /
  flex-direction / アイコン方向）が論理プロパティで書かれているか（`margin-inline-start` 等）。

### G-5. 文字エンコーディング・絵文字
- すべての経路で **UTF-8 / utf8mb4** が前提で、絵文字 / サロゲートペアが文字数カウントや
  比較で破綻しないか（`robust-safety` F-2 と連動）。

## H. API のユーザビリティ（DX / Developer Experience）

API クライアント開発者にとっての「使いやすさ」。
HTTP / GraphQL / gRPC / SDK / Webhook いずれにも適用される観点。

### H-1. 命名の予測可能性（Nielsen #2: Match between system and the real world）
- エンドポイント・フィールド名・パラメータ名が**ドメイン言語**で、推測可能か。
  内部実装由来の名前（`getUserDataInternal` 等）が外部に漏れていないか。
- 命名スタイル（snake_case / camelCase / kebab-case）が**プロジェクト全体で一貫**しているか。
  この差分だけ慣習を破っていないか。

### H-2. デフォルト値の妥当性（Principle of Least Astonishment）
- 省略時の挙動が**最も一般的・最も安全**な選択になっているか。
- ページング上限・タイムアウト・並び順などのデフォルトが、**呼び出し側が驚かない**値か。

### H-3. エラーレスポンスの構造化
- エラーが構造化されている（`application/problem+json` / 独自スキーマ）か（`backend` A-5 / C-3 連動）。
- 同じ性質のエラーは**同じ形式**で返るか（API 横断の一貫性）。

### H-4. ドキュメントとサンプル
- OpenAPI / GraphQL schema / proto がコントラクトとして**最新**か（`functionality` C-1 連動）。
- リクエスト / レスポンスの**実例**（curl / 各言語スニペット）が、コピペで動く形で添えられているか。
- 認証・rate limit・ページング・エラー一覧などの**横断ガイド**が、各エンドポイント説明から
  辿れる位置にあるか。

### H-5. 後方互換とバージョニング
- 破壊的変更が**deprecation 期間 + 通知**を伴うか（`functionality` C-4 / `_reviewer-stance` B-3 連動）。
- バージョニング戦略（URI / ヘッダ）が**明示**され、混在していないか。

### H-6. SDK / クライアント生成物
- 手書き SDK と OpenAPI 生成物が混在して**ドリフト**していないか。
- 言語別 SDK のメソッド名・型名が、その言語の慣習に沿って自然か（強制的な camelCase 押し付け等になっていないか）。

### H-7. 退会・取り消しの DX
- API 側で「**呼び出してしまった**」処理を取り消す経路（DELETE / cancel エンドポイント）が、
  対称形で用意されているか。

## I. 運用者向けユーザビリティ（CLI / 設定 / ログ / ドキュメント）

### I-1. CLI のヘルプと例
- `--help` / `-h` で**何ができるか**と**代表例**が示されるか。
- サブコマンドの命名が予測可能か（動詞 + 名詞 / リソース指向）。
- `--dry-run` / `--yes` / `--quiet` / `--verbose` のような**運用上欲しいフラグ**が揃っているか。

### I-2. 設定の発見可能性
- 環境変数 / 設定ファイルの**全項目**が、デフォルト値と意味とともにドキュメント化されているか。
- 設定エラー時の**メッセージ**が「どの項目が・なぜ・何を設定すべきか」を伝えるか
  （C 節を運用者向けに適用）。
- **dry-run / validate** で実際の適用前に検証できるか。

### I-3. ログ・標準出力の UX
- 通常運用で見るログが**読みやすい量・粒度**か（過剰な debug が本番で垂れ流されていないか）。
  詳細は `reliability` G-1。
- 標準出力（stdout）と標準エラー（stderr）の使い分けが、Unix 慣習に沿っているか
  （成功時はパイプ可能な値のみ stdout、進捗 / 警告は stderr）。
- 終了コードが**意味を持って**設定されているか（成功 0 / 一般エラー 1 / usage エラー 2 等）。

### I-4. アラート文言と次の手
- アラート / エラー通知が「**起きたこと + ランブック / 対応 URL**」を含むか
  （`reliability` H-2 と連動）。
- オンコール担当者が**深夜に見て即座に動ける**情報量か。

### I-5. ドキュメント（README / Changelog / Release Notes）
- README が**最初の 5 分で動かせる**構成か（前提 → install → 最小例 → 次のステップ）。
- Changelog / Release Notes が**ユーザー視点**（何が変わって何が壊れる可能性があるか）で
  書かれているか、内部 commit メッセージの羅列になっていないか。
- 破壊的変更がある場合、**移行手順**が併記されているか。

## 共通: severity 判定の目安（本カテゴリ固有の補足）

`_reviewer-stance.md` A-3 を基本に、本カテゴリでは以下を加味する:

| 区分 | severity の目安 |
|------|----------------|
| タスクが**完遂不能**になる経路 / WCAG Level A 違反で支援技術ユーザーが操作不能 / エラー時に次の手段が無く詰む | blocker |
| タスクは可能だが**著しく困難** / WCAG AA の主要基準違反 / 既存標準パターンから外れた驚き / エラーが「次のアクション」を示せていない | major |
| 標準的な体験向上の余地 / WCAG AAA / 軽い一貫性のずれ | minor |
| 文言・字面・好みの範囲 | nit |

- **「実際にユーザーで再現できる失敗シナリオ」**を持って指摘する。「使いにくそう」だけの主観評価は
  severity を下げる。
- アクセシビリティ違反は、**該当する障害特性のユーザーがいる前提**で評価する（公開 Web は前提アリ、
  社内ツールでは要件次第）。要件不明なら minor + 確認依頼にする。

## 除外・スコープ外

- **要求仕様の充足そのもの・エッジケース挙動** — `functionality` カテゴリ
- **コードの読みやすさ・命名・凝集度** — `maintainability` カテゴリ
- **a11y の実装に密着した部分**（`button` vs `div+onClick` / `htmlFor` / 具体的な ARIA 属性付与）
  — `frontend` H 節（本カテゴリは WCAG 原則・全体方針）
- **エラーメッセージの情報漏洩リスク・enumeration 防止** — `security` F-3
  （本カテゴリは UX 観点で「次のアクションを伝えるか」「実装詳細が混じっていないか」）
- **失敗時のリトライ戦略・冪等性メカニズム** — `reliability` カテゴリ
- **API の HTTP メソッド / ステータスコード / URI 構造** — `backend` A 節
  （本カテゴリは API 全体の DX）
- **lint / 自動チェック可能なルール**（HTML 構文 / aria 属性の構文整合・色コントラスト計測） — 各 CI

# 共通レビュワースタンス — 全 sub-agent 横断の指示

本ファイルはカテゴリ観点ではなく、**全カテゴリの sub-agent に共通して渡される指示**。
担当カテゴリの観点と併せて、findings の severity 判定・suggestion 作成時に参照する。
本ファイルを基準に独立した finding を立てる必要はない。担当カテゴリの findings に「スタンス」として内包して反映する。

---

## A. 修正コスト感覚

### A-1. 一時対応で問題なければ、低コスト案を先に示す
- 1 つの指摘に対して複数の対応策が取り得る場合、**問題解消に必要な最小コストの案を本命として提示する**。
- 「あるべき設計」を本命に推すのは、次のいずれかが具体的に見える時だけ:
  - 暫定対応が**後続の別問題を引き起こす**
  - 暫定対応の**技術的負債が返済されない公算が高い**（履歴・周辺コードから判断できる）
  - 本質的な設計欠陥で、暫定対応でもどのみち同じ箇所を再修正することになる
- 迷ったら**低コスト案を本命にし、恒久対応は「望ましい方向」として併記**する。順序を逆にしない。

### A-2. 提案の粒度を「いま必要な分」に揃える
- 差分の目的を超えたリファクタ・抽象化・責務分離の提案を **suggestion 本体に混ぜない**。
  周辺の改善余地に触れたい場合は、finding の severity を下げて別項として出す。
- YAGNI: 将来の仮想的な拡張のための一般化を勧めない。**3 回目の重複が見えてから DRY 化**でよい。
- 既存コードの命名・構造が最善でなくても、**今回の差分で直す必要がないなら触れない**。

### A-3. severity 判定の目安

| severity | 判定基準 |
|----------|---------|
| blocker  | 本番で顕在化する不具合・データ破壊・セキュリティ事故に直結、または既存機能を壊す |
| major    | 本番で顕在化しうる重大リスク、または設計判断の誤りで低コストで直せる |
| minor    | 局所的な改善余地、放置しても直近では被害がない |
| nit      | 好みの範囲。修正は任意 |

- 「良さそうな改善」を major に吊り上げない。該当しなければ minor か nit。
- 暫定対応でリスクを封じられるなら、**根本原因の指摘は別 finding として minor** で残す。
- **Severity threshold が指定されていても、判定基準を曲げない**。`blocker` / `major` のみを返す指示を受けた場合でも、本来 minor 相当の指摘を blocker / major に吊り上げてはいけない。閾値未満は素直に省略し、件数のみ Notes に書く（収集対象から落ちた事実は隠さない）。

## B. 段階的開発戦略（既存機能に影響する大きい変更）

差分が既存機能の挙動・データ構造・公開 API を変える規模の時、
**ダウンタイムとデータ不整合を起こさずに段階的に出荷できるか**を必ず見る。
単発 PR で完結する小さい内部変更には B-1〜B-5 の多くは適用外。

### B-1. 単発デプロイで完結させていないか（Expand & Contract）
- スキーマ変更・API 互換性変更・配信形式変更など、**旧実装と新実装が同時に動く瞬間**が生まれる変更で:
  - 旧クライアント / 旧プロセス / 旧レコードが残った状態で新コードが正しく動くか
  - 新クライアント / 新プロセスが、旧レコード / 旧レスポンスを受け取っても落ちないか
- **Expand and Contract**（追加 → 両対応 → 切り替え → 旧削除）の各フェーズが独立した PR / デプロイに
  分解できる構造になっているか。1 PR に全フェーズが同居していないか。

### B-2. データマイグレーション
- 単発デプロイで破壊的スキーマ変更を入れていないか:
  - NOT NULL 列の**無デフォルト**追加、**既存列の rename / drop**、**型変更**、**index 付け替え**
  - 段階化（add → backfill → start-writing-new → stop-writing-old → drop-old）されているか
- **データ移行スクリプトが冪等で、途中で再実行しても安全か**
- **ダウンタイムを伴うロック**（長時間 `ALTER TABLE` / フルテーブルスキャン等）を含んでいないか。
  巨大テーブルには online migration / chunked 処理 / shadow table 方式を検討したか
- **新旧データが共存する期間**を読み出しコードが想定しているか
  （旧 null 値 / 旧フォーマットに対するフォールバックが入っているか）

### B-3. 後方互換性
- **公開 API（REST / GraphQL / gRPC / イベントスキーマ）**: 既存クライアントを壊していないか
  - フィールド削除 / リネーム / optional → required / enum 値追加（closed 扱いの受信側を壊す）
  - **追加 → deprecation 期間 → 削除** の段階に分解できているか
- **メッセージキュー / イベント**: 旧フォーマットのメッセージが inflight にある状態で新 consumer が読めるか、
  その逆（新フォーマットを旧 consumer が読む）も成り立つか
- **内部関数の破壊的変更**でも、デプロイ境界（別プロセス / 別サービス / キャッシュ / ジョブキュー）を
  跨ぐ場合は同じ配慮が必要

### B-4. ロールアウト / ロールバック経路
- **Feature Flag** で新挙動を隠せるか。flag OFF で旧挙動が完全に維持されるか。flag 切り替えが原子的か
- **段階的ロールアウト**（canary / percentage / ring）で異常検知 → 即ロールバックできるか
- **ロールバック経路が成立しているか**。旧コードへのリバートだけで元に戻るか、
  それとも**データが進んでしまって戻せない状態**を作っていないか
  （forward-only マイグレーションの扱いが明示的に設計されているか）

### B-5. 可観測性（新旧共存期間）
- 新旧共存中に、**どちらのパスを通ったか**を識別できるログ / メトリクス / トレースが仕込まれているか
- 移行完了判定（= 旧コード削除に進めるか）の**判断材料**がダッシュボードで見えるか
  （新クライアント率・旧データ残件数・失敗率 など）
- 異常時に**どの段階で詰まったか**を切り出せるか

### B-6. 提案の出し方
- 「大きい変更が 1 PR に詰め込まれている」ことを指摘する時は、
  **具体的なフェーズ分割案**（追加 / 両対応 / 切り替え / 削除）を suggestion に書く。指摘だけで代案なしにしない
- 既にフェーズ分割されていて十分安全と判断したら、B 系の指摘はスキップしてよい

## 各カテゴリ sub-agent への適用方法

- まず担当カテゴリの観点で findings を出したあと、**各 finding の severity・suggestion を A / B に照らして見直す**
- A に反する過剰な提案は縮小する。B に関わる差分なら、**担当カテゴリの視点から** B-1〜B-5 を追加点検する
  - 例: `backend` なら B-2 をドメインロジック / ORM レイヤから、
    `infrastructure` なら B-4 を IaC / デプロイパイプラインから、
    `frontend` なら B-3 を API クライアント / キャッシュ整合性から見る
- 本ファイルの内容を独立カテゴリとして扱わず、担当カテゴリの findings に内包して反映する

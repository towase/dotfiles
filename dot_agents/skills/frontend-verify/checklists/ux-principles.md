# ux-principles 観点 — デザイン 4 原則（近接・整列・反復・強弱対比）

ロビン・ウィリアムズ『ノンデザイナーズ・デザインブック』の 4 原則に沿って、実装が情報を正しく伝えているかを判定する。

`frontend-verify` Phase 3-2 で参照される。

**判断材料**: 各 viewport のフルページスクショ（**実画像**）。DOM だけで判断しない。
**チェック対象**: 今回の差分で追加・変更された画面を最優先。既存画面と比較しての一貫性も見る。

---

## A. 近接 (Proximity)

> **関連するものは近く、関連しないものは遠く。**

### A-1. ラベルと入力フィールド
- ラベルがどの入力に紐づくか視覚的に明確か（距離・線・グルーピング）
- ラベルが「上の入力」と「下の入力」のどちらに属するか曖昧になっていないか
- 補助テキスト（help text）が入力フィールドに近接しているか、別フィールドに紛れていないか

### A-2. 関連項目のグルーピング
- フォーム内で関連項目（住所 ZIP / 都道府県 / 市区町村）が 1 グループとして近接配置されているか
- 関連しない項目との間に**明確な間（マージン）**があるか
- 「項目間の余白が全部同じ」になっていて、グルーピングが視覚的に消失していないか

### A-3. カード / セクションの内側余白
- カード内の要素は近く、カード同士は離れているか（**内側余白 < 外側余白** の関係）
- カードの境界（border / shadow）に頼らず、余白だけでもグループが認識できるのが理想

### A-4. ボタン群
- 「保存 / キャンセル」のような対になるボタンは近接しているか
- 同じ場所に複数の独立アクションが詰め込まれていて、誤クリックを誘発していないか
- プライマリと破壊アクション（削除など）が誤って近接していないか（誤爆防止）

### A-5. アイコンとテキスト
- アイコン + テキストのペアで、アイコンが間違って隣のテキストに引き寄せられていないか
- アイコンとラベルの間隔が、ペア間の間隔より十分狭いか

---

## B. 整列 (Alignment)

> **画面内のすべての要素は、必ず別の要素と意図的に揃える。**

### B-1. 主要な揃え軸
- 画面全体で**左揃え / 右揃え / 中央揃えのいずれかが支配的**か（無秩序な混在になっていないか）
- フォームのラベル・入力・補助テキストの**左端が揃っている**か
- セクション見出しと本文の左端が揃っているか

### B-2. 中央揃えの濫用
- 中央揃えは「強調したい単独要素」のみに使われているか
- 長いリスト / 段落本文を中央揃えしていないか（読みにくくなる）
- 行ごとの長さがバラバラのテキストを中央揃えしていないか

### B-3. ベースライン
- 並んだボタン同士、隣接するアイコン + テキストの**垂直中心が揃っている**か
- アイコンサイズと隣接テキストサイズが釣り合っているか（極端にアイコンが大きい / 小さい）
- 上下に並ぶラベルとバッジが垂直中央で揃っているか

### B-4. テーブル / リスト
- カラムの右端が揃っているか（特に**数値カラムは右揃え**）
- リストアイテムの左端が揃っているか（インデントが意味なくバラバラになっていないか）
- 列内のアラインが意味的に揃っているか（金額・日付・タグ）

### B-5. グリッド整合
- カード / セクションの幅・位置が共通グリッドに乗っているか
- 1 つだけ幅 / オフセットが違う「迷子の要素」がないか

---

## C. 反復 (Repetition)

> **同じ役割の要素は同じ見た目で繰り返す。**

### C-1. ボタンスタイル
- プライマリ / セカンダリ / 破壊（destructive）の各ボタンが、画面間で同じ見た目か
- 今回追加した画面 / コンポーネントだけが浮いていないか（独自の色 / 角丸 / 影 / フォントサイズ）

### C-2. 見出しスタイル
- h1 / h2 / h3 のサイズ・色・余白が一貫しているか
- 同じ階層の見出しが画面によって違うスタイルになっていないか

### C-3. アイコンセット
- 同じ意味（編集 / 削除 / 設定 / 追加）のアイコンが画面間で一貫しているか
- 異なるアイコンライブラリ（Material / Lucide / Heroicons など）が混在していないか
- アイコンの線の太さ・スタイル（filled / outline）が混在していないか

### C-4. スペーシング
- カード間 / セクション間の余白が**一貫したスケール**（4 / 8 / 16 / 24 / 32px 等）に乗っているか
- 「ここだけ 13px、ここだけ 17px」のような非標準値が混じっていないか

### C-5. インタラクション
- hover / focus / active / disabled の表現が画面間で一貫しているか
- 同じ役割のボタンで hover 時のフィードバックが違っていないか

### C-6. レイアウトパターン
- リスト表示 / カード表示 / 詳細パネルなど、同種コンテンツの配置パターンが一貫しているか
- 同種のフォームで、submit ボタンの位置（右下 / 左下 / 上）が画面ごとに揃っているか

---

## D. 強弱対比 (Contrast)

> **重要なものは目立たせ、それ以外は引っ込める。**

### D-1. アクションの優先順位
- **プライマリアクション**（最も推奨する操作）が視覚的に最も目立つか
- セカンダリ / 破壊アクションが、誤って主役にならないトーンになっているか
- **複数のプライマリボタンが並んで**「どれを押せばいいか分からない」状態になっていないか（プライマリは原則 1 つ）
- 破壊アクション（削除）が赤系で、明確に区別されているか

### D-2. 見出しと本文のジャンプ率
- 見出しと本文のフォントサイズ差が、情報構造を伝えるのに十分か
- すべて同じサイズで「のっぺり」していて、何が見出しか判別できなくなっていないか
- 階層が深い場合（h1 / h2 / h3 / h4）、それぞれが視覚的に区別できるか

### D-3. 色のコントラスト
- テキスト / 背景のコントラスト比が十分か（WCAG AA: **4.5:1** / 大文字 3:1）
- 薄いグレーのプレースホルダ / 補助テキストが読めるレベルか
- 色だけに情報を依存していないか（色覚多様性配慮 — エラーは赤色 + アイコン + テキスト等の冗長性）

### D-4. 余白による強弱
- 重要なエリアの周囲に余白を確保して、注目を集めているか
- すべてが詰まりすぎていて優先度が伝わらない状態になっていないか
- CTA（Call to Action）周辺の余白が十分にあるか

### D-5. 状態の差
- disabled / loading / error / 通常状態 が視覚的に明確に区別されるか
- disabled が「ただ薄い」だけで、見落とされる可能性がないか（カーソル `not-allowed` / 透明度 + 質感変化）

### D-6. フォントウェイトと色
- 重要情報が太字 / 濃い色で、補助情報が細字 / 淡い色で表現されているか
- 太字の濫用で「全部強調 = 何も強調されていない」状態になっていないか

## severity の目安

| 状態 | severity |
|------|----------|
| 情報構造が崩壊してユーザーが操作対象を判別できない / プライマリアクションが見えない / WCAG AA コントラスト未満で読めない | blocker |
| 4 原則の重大な違反（中央揃え濫用 / プライマリボタン乱立 / 既存画面から完全に浮いた独自スタイル） | major |
| 軽い一貫性のずれ / 局所的な余白の不揃い / 個別箇所のジャンプ率不足 | minor |
| 好みの範囲 / 細部の調整余地 | nit |

---

## 除外・スコープ外

- **WCAG 観点全般**（キーボード操作 / aria / 拡大耐性）— 本 skill `common-issues.md` + `review-multi-agent` の `usability.md` F 節
- **コンポーネント構造 / props 設計** — `review-multi-agent` の `frontend.md`
- **コピー / 文言の言い回し** — 本 skill ではトーン違反のみ拾い、文言改善は別途

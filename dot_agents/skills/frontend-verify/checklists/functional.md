# functional 観点 — 機能性チェック

`frontend-verify` の Phase 3-1 で参照。`chrome-devtools` MCP で実画面を操作して、機能が動くかをチェックする観点リスト。

**チェックの基本姿勢**: ボタンを押すだけで終わらず、**前後の状態 / コンソール / ネットワーク** を毎回観察する。

---

## A. 状態の網羅（loading / empty / error / success）

Nielsen #1: Visibility of system status。新しい画面 / 新しい操作で 4 状態すべてが意図的に設計されているかを見る。

### A-1. Loading
- ボタンクリック直後に loading 表示（スピナー / disabled / スケルトン）があるか
- 長時間（>3 秒）処理に進捗 or キャンセル経路があるか
- **確認手順**: `click` 直後に `take_screenshot` → 1 秒後に再撮影、視覚的変化があるか

### A-2. Empty
- データが 0 件のとき、空白ではなく「まだ X がありません。Y から作成できます」のような手引きが出るか
- **確認手順**: 可能ならクエリパラメータ / 空 ID で空状態を再現

### A-3. Error
- 失敗時のメッセージが「現象」+「**次のアクション**」を伝えるか（CLAUDE.md 規約）
- フィールド単位エラーは該当フィールド近辺に出るか（トーストだけでフィールド特定できない経路になっていないか）
- ❌ 「エラーが発生しました」「Bad Request」「500 Internal Server Error」
- ⭕ 「メールアドレスの形式が正しくありません。`@` を含めてください」
- **確認手順**: ネットワークを切る or `evaluate_script` で fetch をモックして 4xx / 5xx を発生させる

### A-4. Success
- 完了後にトースト / 画面遷移 / インライン表示などの**完了フィードバック**があるか
- 「ボタンを押したのに何も起きないように見える」状態になっていないか

---

## B. 主要フローの実行

### B-1. フォーム送信
- `fill_form` → `click`（送信）→ `wait_for` で結果を観測
- **必須未入力**で送信 → エラー表示確認
- **不正フォーマット**入力 → エラー表示確認
- **正常入力** → 成功表示確認
- バリデーション発火タイミング（onChange / onBlur / onSubmit）が、入力速度に対して妥当か（タイプ中に真っ赤になって責められていないか）

### B-2. 二重送信防止
- 送信ボタンが送信中 disabled になるか
- 連打して `list_network_requests` を観察し、二重リクエストが飛んでいないか
- 楽観的更新（optimistic UI）使用時、失敗時の戻しが破綻していないか

### B-3. ナビゲーション
- 主要リンク / ボタンを 1〜2 本踏んで遷移確認
- ブラウザ「戻る」（`press_key` で Alt+← or History API）で状態が破綻しないか
- 直接 URL アクセス（`navigate_page` で再度 URL を打つ）でも同じ画面が見えるか

### B-4. プルダウン / Select
- 開く → 選択 → 反映 を確認
- キーボード操作（↑↓ Enter / Esc）でも操作可能か（`press_key` で検証）
- 選択肢の長文ケースは `common-issues.md` A 節も併せて見る

### B-5. モーダル / ダイアログ
- 開く → 中身が見える → 閉じる
- 背景クリック / Escape キーで閉じるか
- 開いた状態で**背景スクロールが抑制**されるか（`evaluate_script` で `document.body.style.overflow` を確認）
- フォーカスがモーダル内にトラップされるか（Tab を押し続けてフォーカスが外に出ないか）

### B-6. インクリメンタル機能（検索 / フィルタ / ソート）
- 入力 → 結果が更新される
- デバウンス / スロットリングが効いて過剰なリクエストが飛んでいないか（`list_network_requests` で連続呼び出しを観察）
- 0 件ヒット時の表示（B-A2 Empty と接続）

---

## C. コンソール / ネットワーク

### C-1. コンソールエラー
- 初期ロード時に red エラー（`error` level）がないか
- 主要操作後に新規エラーが出ないか
- warn は無視可だが、明らかにフレームワーク由来でない warn は finding 化検討

### C-2. ネットワーク失敗
- 4xx / 5xx の失敗リクエストがないか
- 失敗時に UI が破綻していないか（白画面 / スピナー固まり / 二重通知）
- レスポンスが想定形（JSON 等）と一致しているか（`get_network_request` でボディ確認）

### C-3. リクエスト数
- 1 操作で過剰なリクエスト数になっていないか（N+1 的なパターン）
- 同じエンドポイントに重複リクエストが飛んでいないか

---

## D. 入力保護・離脱対策

### D-1. 入力途中の保護
- フォーム入力途中で `navigate_page` で別ページへ → 戻ったときに入力内容が残っているか / 失われる場合は事前 confirm が出るか
- 仕様として「残らなくて良い」場合は明示的にユーザー要件と照合

### D-2. アクセシブル認証（WCAG 3.3.8 AA）
- 認証フローで、記憶 / 認知タスクへの依存だけが残っていないか（パスワードマネージャ自動入力 / 生体認証 / マジックリンク等の代替）

---

## severity の目安

| 状態 | severity |
|------|----------|
| 主要タスクが完了できない / 白画面 / 致命的エラー / 二重送信で副作用が二重発火 | blocker |
| タスクは完了できるが手順が破綻 / コンソールに red エラー / 失敗時に次の手段が示されない | major |
| 状態フィードバックの軽微な不足 / 表示の小さなずれ / 警告系 console log | minor |
| 文言・字面・好み | nit |

## 除外・スコープ外

- **コードの読みやすさ / 設計** — `review-multi-agent` の `frontend` / `maintainability`
- **デザイン 4 原則の視覚評価** — 本 skill `ux-principles.md`
- **a11y の WCAG 違反全般** — 本 skill `common-issues.md` E / G 節（フォーカス・タップ）と部分的に重複、深く見るなら `review-multi-agent` の `usability.md` F 節

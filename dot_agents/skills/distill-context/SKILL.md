セッションを振り返り、**次回以降の AI 開発の判断精度を上げる情報だけ**を抽出して、永続コンテキスト（AGENTS.md / CLAUDE.md / skills / decision-log など）へ反映する。

## 核心原則

```
記録するのは「作業内容」ではなく「次回の判断を良くする情報」
```

- 「今日はログイン画面を直した」は価値が低い（git が記録する）。
- 「このプロジェクトではフォーム状態は react-hook-form に寄せ、独自 useState は避ける」は価値が高い（次回の判断を変える）。

**全部は残さない。** 全部残すとコンテキストが腐り、かえって AI の判断が鈍る。承認前に自動で追記しない（数週間で「プロンプトのヘドロ」になる）。情報量を増やすより、次回の判断精度を上げることを優先する。

## 引数

$ARGUMENTS

任意。振り返りで特に注目してほしい論点や対象範囲があれば指定する（例: 「テスト周りの手戻りを中心に」）。空なら今セッション全体を対象にする。

## 厳選の判断基準

| 抽出した知見 | 分類 | 反映先の目安 |
| --- | --- | --- |
| 3 回以上再利用されそうな前提・制約 | 保存候補 | project-context / AGENTS.md |
| 設計・技術の意思決定（なぜ A ではなく B か） | 保存候補 | decision-log |
| 手順化できる繰り返し作業 | 保存候補 | skill 新規作成 |
| AI が誤解した・人間が補足した → 再発防止ルール | 保存候補 | AGENTS.md / CLAUDE.md（または feedback メモリ） |
| 1 回限りの作業内容 | 捨てる | session-notes 止まり（長期コンテキストに入れない） |
| 古くなりやすい情報・具体的すぎるファイル名 | 捨てる | 入れない |
| 好み・その場の判断にすぎないもの | 捨てる | 入れない |

## プロセス

### 1. 振り返る

今セッションから次を洗い出す（作業ログではなく「次回に効くもの」の種を探す）:

- 今回実装・調査したことの要点（3〜7 行）
- 詰まった点・手戻り・AI が誤解した点・人間が補足説明した点
- 次回も同じパターンで使えそうな作業手順
- プロジェクト固有の判断・制約・前提

必要なら `git log`・差分・編集したファイルを見て事実を補強する。推測で書かない。

### 2. 厳選する

上の判断基準で各知見を「保存候補 / 捨てる」に振り分け、保存候補をさらに反映先（前提・設計判断・手順・再発防止ルール）で分類する。除外するもの（一度きり・古くなりやすい・具体的すぎるファイル名・好み）はここで落とす。

### 3. 反映先を検出する

カレントプロジェクトとグローバルの両方で、反映先になりうるファイルの有無を調べる。**勝手に新規作成しない**——存在しない反映先（decision-log.md / project-context.md など）が必要な場合は「新規作成」として提案に含め、承認を得てから作る。

- プロジェクト: `AGENTS.md` / `CLAUDE.md` / `decision-log.md` / `docs/` / `project-context.md` / `.claude/skills/` / `README`
- グローバル: `~/.claude/CLAUDE.md` / `~/.claude/skills/`（dotfiles 管理下なら、そのリポジトリの CRUD/反映ルールに従う）
- 環境にメモリ機構があれば feedback / project メモリも候補

ルール・前提は、その内容がプロジェクト固有ならプロジェクト側、全プロジェクト共通ならグローバル側へ振り分ける。

### 4. コンテキスト更新パッチを HTML で提示する

`/tmp/distill-context-<topic>.html` に書き出し、続けて `open <filepath>` を実行する（CLAUDE.md のコミュニケーション規約準拠。ライトモード固定。`AskUserQuestion` は使わない）。コンソールには書き出したパスのみ伝える。

HTML に含める:

- **今回のサマリー**（3〜7 行）
- **コンテキスト更新パッチ**——各項目に通し番号を振り、ユーザーがコンソールで「1,3 採用 / 2 保留」のように番号で回答できるようにする。3 分類で出す:
  - **すぐ追加すべき**: 項目ごとに〔理由 / 反映先（ファイルパス）/ 追加内容（そのまま貼れる Markdown）/ AI 推奨度〕
  - **保留**: 〔理由 / 採用する条件〕
  - **追加しない**: 〔理由〕（捨てた根拠を明示し、隠れた切り捨てを残さない）
- **次回セッション開始プロンプト**（任意）: この続きから始める短いプロンプト

skill 化候補がある場合は、`### skill: <名前>` の見出しで〔いつ使うか / 手順 / 注意点 / 完了条件〕の骨子も併せて提示する。

### 5. 採用/保留/破棄を待つ

ユーザーが番号で採否を返すまで反映しない。回答が曖昧なら確認する。

### 6. 採用分だけ反映する

採用された項目のみ実ファイルへ反映する:

- **ルール / 前提**: AGENTS.md / CLAUDE.md の該当セクションへ Edit で追記（既存の節構成・文体に合わせる）
- **設計判断**: decision-log にエントリ追加。書式は `## YYYY-MM-DD: <タイトル>` ＋〔決定 / 背景 / 理由 / 代替案 / 今後見直す条件〕
- **手順（skill 化）**: `skills/<名前>/SKILL.md` を新規作成。反映先リポジトリに skill の CRUD/反映ルール（chezmoi など）があればそれに従う
- **一度きり / 作業ログ**: 長期コンテキストには入れず、必要なら session-notes に残すか破棄する

反映後、何をどこへ反映したか（と保留・破棄の件数）を簡潔に報告する。

## やらないこと

- 承認前に永続コンテキストへ自動追記する（ヘドロ化の原因）
- 振り返りの全項目を記録する（厳選を放棄する）
- 作業ログ・履歴的記述・一度きりの暫定対応を長期コンテキストに入れる
- 古くなりやすい情報・具体的すぎるファイル名・好み・その場の判断を反映する
- 反映先ファイル（decision-log / project-context など）をユーザー承認なしに新規作成する
- HTML 提示の代わりに `AskUserQuestion` を使う（CLAUDE.md のコミュニケーション規約）

## 使い分け

- 実装しながら判断ログを残したい → `implement-with-notes`
- 既存ドキュメントが冗長で圧縮したい → `compact-docs`
- skill / プロンプトの指示文そのものを実測で改善したい → `empirical-prompt-tuning`
- セッションを振り返って次回のためにコンテキストを蒸留・反映したい → このスキル

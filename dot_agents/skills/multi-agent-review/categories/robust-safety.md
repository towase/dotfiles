# robust-safety 観点（優先順位: 3）

## このカテゴリで見るもの
**コードレベルでの堅牢性・安全性**。reliability がシステム全体の失敗時挙動を見るのに対し、こちらは
「コードが不正状態を作らない / 不正状態が観測可能な形で失敗する」ためのミクロな設計判断を扱う。
具体的には、**型による不正状態の不可表現化**、**境界での検証と内部の信頼**、**fail-safe デフォルト**、
**並行実行下のデータ安全**、**リソース解放の確実性**、**数値・時刻・文字列の罠**、**列挙網羅性** など。

機械的に検出できる範囲（型エラー / 未使用変数 / null safety の言語機能）は linter / 型システムに委ね、
ここでは「**人間の判断が必要な堅牢性のトレードオフ**」を見る。

## 観点構成
- A. 入出力境界での検証と内部の信頼
- B. 型による不正状態の不可表現化
- C. fail-safe デフォルト
- D. 並行・並列での安全性
- E. リソース管理（解放の確実性）
- F. 数値・文字列・時刻の罠
- G. 列挙の網羅性
- H. 例外・エラー伝播の設計
- I. 不変性と防御的コピー

---

## A. 入出力境界での検証と内部の信頼

### A-1. 境界の特定と検証
- 外部入力（HTTP リクエスト / キュー / ファイル / 外部 API レスポンス / DB から読んだ値 / 環境変数）が、
  **境界で 1 度だけ** 検証され、検証済みの型に変換されているか。
- 境界以降の内部関数で、同じ値を **再 validate** する重複検証が散らばっていないか。
- 境界で検証を通過した値が、その後 **「正しい型」として表現** されているか
  （例: `string` ではなく `Email` / `UserId` のような nominal/branded 型）。

### A-2. 信頼境界の越境
- 「内部だから安全」と思って `unknown` / `any` / `interface{}` / `JSON` を素通りさせていないか。
- 信頼境界をまたぐデータ（外部システムから戻ってきた値、永続化から読んだ値）が、
  **アプリ内型の不変条件を満たすかが再検証** されているか。

### A-3. 部分関数の網羅化
- 「特定条件下でのみ正しく動く」関数（部分関数）が、入力型の段階でその条件を表現しているか。
  例: `parseInt(string): number` ではなく `parseInt(string): number | ParseError`。

## B. 型による不正状態の不可表現化

### B-1. nullable / optional の意味づけ
- `nullable` / `optional?` / `Maybe` / `Option` の使い分けが「値の不在の意味」と一致しているか。
  「未取得」「未設定」「該当なし」「N/A」を一律 `null` にしていないか。
- DB の `NULL` とアプリの `null` / `undefined` の意味が同一になっているか、変換層で正規化されているか。

### B-2. bool フラグの組合せ爆発
- 複数の bool フラグで状態を表現している箇所で、**起こり得ない組合せ** が型として許されていないか。
  状態 enum / sum type / discriminated union への置き換えが検討されているか。
- 例: `isLoading` + `isError` + `data` の 3 値を、`Idle | Loading | Success | Error` の sum type に畳めないか。

### B-3. 状態遷移の表現
- ライフサイクルを持つオブジェクト（注文・ジョブ・接続）で、**取り得ない遷移が型レベルで防がれている** か。
- 「`status === 'paid'` のときだけ `paid_at` がある」のような暗黙の依存を、型で結合できないか
  （discriminated union / sealed class / sum type）。

### B-4. プリミティブ過用
- `string` / `number` で意味の異なる値（UserId と OrderId、円と USD）を区別なく扱っていないか。
  branded / newtype 化のコストが見合うホットパスかが検討されているか。

---

## C. fail-safe デフォルト

### C-1. 失敗時の方向性
- 設定読み込み失敗・パース失敗・タイムアウト時の既定動作が、**安全側（拒否 / 機能停止）** に倒れているか。
  特に **認可・課金・送信** に関わる処理で「失敗時に許可してしまう」になっていないか。
- feature flag のフェッチ失敗時に「機能 ON」にフォールバックしていないか
  （UX 都合で ON が望ましい場合は、その判断が明示されているか）。

### C-2. 例外握り潰し
- `catch (e) {}` / `try ... except: pass` / `result.unwrap_or_default()` 的な記述が、
  **意図された無視** か **不注意な握り潰し** か区別できる形で書かれているか。
- ロギング・監視通知が抜け落ちていないか（reliability の観測性とは別に、コードレベルで失敗が見える形か）。

### C-3. パニック・unwrap・非 null 表明
- `!` / `unwrap()` / `expect()` / `as!` / `!!` が、**真に到達不能な前提** にのみ使われているか。
  業務入力に依存する箇所で使っていないか。
- パニックがプロセス全体を落とす言語（Go / Rust 等）で、recover / catch_unwind が必要な箇所が抜けていないか。

### C-4. デフォルト値の安全性
- 構造体・レコード・DTO のデフォルト値が、**安全側** に倒れているか
  （`isPublic: true` がデフォルトになっていないか、`retries: 0` が望ましい挙動か など）。

---

## D. 並行・並列での安全性

### D-1. 共有可変状態の保護
- スレッド / goroutine / async タスク間で共有される可変状態が、適切な同期機構（mutex / atomic / channel / actor）で保護されているか。
- 「グローバル変数 + 単発代入」のように見えても、初期化レースが起きていないか。

### D-2. レースコンディション
- 「読んでから書く」「存在チェックしてから挿入する」パターンが、適切なトランザクション境界・ロック・
  upsert 構文で **アトミック** になっているか。
- 同一リソースに対する並行更新で、**最後の書き込みが勝つ** が妥当な業務要件か、楽観ロックが必要か。

### D-3. キャンセル伝播
- 長時間処理 / 外部 I/O が、**キャンセル / タイムアウト** を受け取って中断できるか
  （`context.Context` / `AbortSignal` / `CancellationToken` 等の伝播）。
- キャンセル後にリソース解放・部分結果の扱いが整理されているか。

### D-4. 非同期エラーの扱い
- Promise / Future / Task の **未捕捉拒否（unhandled rejection）** が起き得る経路がないか。
- `Promise.all` / `Promise.allSettled` の使い分けが、部分失敗時の業務要件と一致しているか。

---

## E. リソース管理（解放の確実性）

### E-1. 解放の保証
- ファイル・接続・ロック・トランザクション・サブスクリプション・タイマーが、
  **正常系・異常系の両方** で確実に解放されているか
  （`try-finally` / `defer` / `using` / `with` / RAII）。
- 早期 return や例外パスで解放が漏れていないか。

### E-2. 解放順序
- 依存関係のあるリソース（接続 → ステートメント → 結果セット）の解放順序が逆順で正しいか。
- 解放処理自体が例外を投げ得る場合の処理が考慮されているか。

### E-3. リーク傾向
- イベントリスナ・オブザーバ・サブスクリプション・タイマー・DB 接続プールが、
  オブジェクトの寿命と一致して解除されるか。
- 長寿命オブジェクトに短寿命オブジェクトの参照を保持して、GC が回収できなくなっていないか。

---

## F. 数値・文字列・時刻の罠

### F-1. 数値
- 整数オーバーフロー / アンダーフロー（特に符号付き ↔ 符号なし変換、`int32` 上限超え）が起こり得る箇所で、
  検査・型選択（`int64` / `BigInt` / `decimal`）が適切か。
- 浮動小数点で **等価比較** をしていないか、累積誤差が問題になる集計を float でやっていないか。
- 金額・税率など精度要件のある計算で `float`/`double` を使っていないか（`decimal` / 整数最小単位）。

### F-2. 文字列
- 文字列のバイト数と文字数を混同していないか（UTF-8 のマルチバイト、サロゲートペア、絵文字、
  結合文字列でのインデックス計算）。
- 大文字小文字変換・比較がロケール依存で壊れる経路がないか（Turkish I 問題、ドイツ語 ß など）。
- 正規化（NFC / NFD）の選択が、保存・比較・検索で一貫しているか。
- 改行コード（CRLF / LF）・BOM・末尾空白・ゼロ幅文字の混入対策が必要な経路で取られているか。

### F-3. 時刻・日付
- タイムゾーンを **UTC で保持し境界で変換** する原則に従っているか。`Date` / `DateTime` / `Instant` /
  `LocalDateTime` の使い分けが意図通りか。
- 夏時間切り替え時の「存在しない時刻 / 二重に存在する時刻」が業務に影響する経路で考慮されているか。
- 「日付の差」「N 日後」を素朴な秒数加算でやっていないか（カレンダー演算 vs 経過時間演算）。
- 期間の閉開区間（`[start, end)` / `[start, end]`）の扱いが API・DB・UI で一貫しているか。

## G. 列挙の網羅性

### G-1. switch / match の網羅
- enum / sum type に対する分岐が、コンパイラ支援で **網羅性検査** されているか
  （`never` assertion / `default` で fallthrough / `match` の exhaustive チェック）。
- `default` 句で「将来追加された値」を **silently 無視** していないか。fail-safe にエラーへ倒すべき経路で
  そうなっているか。

### G-2. 値追加への耐性
- enum 値追加時に、対応漏れの分岐がコンパイル時に検出できるか、それともランタイムまで気付けないか。
  気付けない場合、テスト・契約で補えているか。

### G-3. オープン enum / クローズ enum
- 外部システムから来る enum を **クローズに扱って** いないか（未知値で例外を投げる前に、
  「未知値を表す型」を持つべきか検討されているか）。

## H. 例外・エラー伝播の設計

### H-1. キャッチ層
- 例外を **どの層で** キャッチするかが整理されているか（境界層で集約 / 各層で wrap / fail fast）。
- 同じ例外を複数層で重複キャッチして文脈を失っていないか。

### H-2. wrap / re-throw
- 低レベル例外（DB エラー等）を上位層で扱う際、原因情報を保ちつつドメイン例外に **wrap** されているか。
  情報を失う catch（メッセージだけ string 化して再投げ）になっていないか。

### H-3. 値ベースのエラー伝播
- `Result<T, E>` / `Either` / 戻り値タプル方式の言語で、エラーを **無視できない** 形（型で強制）に
  なっているか。`?` / `try!` / `if err != nil` の扱いが規約と一致しているか。

### H-4. recoverable / unrecoverable の分離
- 業務上回復可能なエラーと、プログラマの前提崩壊（assertion 違反）を **同じ catch で扱っていない** か。
  後者は早期失敗で観測されるべき。

---

## I. 不変性と防御的コピー

### I-1. 不変性のデフォルト
- 値オブジェクト・設定・DTO が **デフォルトで不変** になっているか。
  可変が必要な箇所だけ明示的に可変にする方針か。

### I-2. 共有参照の罠
- コレクション・オブジェクトをそのまま外部に返して、呼び出し元での変更が内部状態を破壊し得る経路がないか。
- 防御的コピー or イミュータブル型のいずれで担保するかが一貫しているか。

### I-3. 凍結・読み取り専用
- API として「読み取り専用」を表現する際、言語機能（`readonly` / `Final` / `frozen`）と契約が一致しているか。

---

## 除外・スコープ外
- 認証・認可・暗号・サプライチェーンの脆弱性そのもの（`security` カテゴリ）
- システム全体の失敗時挙動・リトライ戦略・観測性（`reliability` カテゴリ）
- スループット・レイテンシ・メモリ効率（`performance` カテゴリ）
- 機能要件の充足とエッジケース動作（`functionality` カテゴリ）
- 静的に検出可能な型エラー・lint 違反（各プロジェクトの linter / 型チェッカに委譲）

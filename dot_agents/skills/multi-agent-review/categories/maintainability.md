# maintainability 観点（優先順位: 6）

## このカテゴリで見るもの
責務分離、ドメイン概念のモデリング、テスト容易性を中核に、可読性・命名・凝集度・依存方向・重複・ドキュメントを含めて「人間の判断が必要な設計上の健全性」を見る。
※静的検査可能なルール（lint / format / ast-grep）はこのカテゴリではなく linter 側で担保する方針。

## 主要チェックリスト

### 1. 責務が適切に分離できているか
- 1 つのモジュール / クラス / 関数が「複数の変更理由」を抱え込んでいないか（単一責任）
- 抽象度のレベルが 1 つの関数内で混在していないか（高レベルの手続きと低レベルの実装詳細が同居していないか）
- レイヤ・関心（UI / アプリケーション / ドメイン / 永続化 / I/O）の境界が差分の中で滲んでいないか
- 副作用（I/O・時刻・乱数・グローバル状態の更新）と純粋ロジックが分離されているか
- 凝集度：一緒に変わるものが一緒に置かれているか／別々に変わるものが分離されているか
- 依存方向が一方向（上位 → 下位、具象 → 抽象）になっているか、循環依存や逆流が混入していないか

### 2. エンティティが適切な概念として表現できているか
- ドメイン上意味のある値が primitive（string / int / map 等）や匿名オブジェクトのまま露出していないか（primitive obsession / 値オブジェクト化の漏れ）
- 同じ概念が複数の場所で別名・別型・別形式で表現されていないか（概念の重複・分裂）
- エンティティの不変条件（invariant）が生成・更新パスでコードとして保証されているか、呼び出し側の善意に依存していないか
- ライフサイクルや状態遷移が型・状態機械で表現されているか（「あり得ない状態」を型として表現できない状態になっていないか）
- 命名がドメインの言葉（ubiquitous language）と一致しているか、実装都合の名前に寄っていないか
- データと振る舞いの置き場所が適切か（貧血ドメイン / 手続き的ユーティリティに業務ロジックが漏れていないか）

### 3. テストがしやすい構造になっているか
- テスト対象が外部依存（DB / ネットワーク / 時刻 / 乱数 / ファイル / 環境変数）から切り離せる構造になっているか（依存性注入・境界の明示）
- 暗黙のグローバル状態・シングルトン・静的メソッド経由の隠れた結合がないか
- テストを書くためだけに prod コードの可視性を緩める（public 化、friend、export-for-test）必要が生じていないか
- テストが内部実装ではなく「振る舞いの契約」を検証できる API 境界になっているか（リファクタで壊れないテストを書けるか）
- セットアップの巨大さ・モックの数がコードの結合度の高さを示唆していないか
- 失敗時の原因特定が容易な粒度で分割できるか（1 テスト = 1 振る舞い）

### 4. 補助的な観点
- 命名：名前から意図と制約が読み取れるか、読み手が文脈で推測する必要がないか
- 重複：偶発的重複（たまたま似ている）と本質的重複（同一概念）を区別しているか（DRY の誤適用になっていないか）
- ドキュメント／コメント：コードで表現できない「なぜ」「前提」「制約」だけが残り、「何をしているか」の重複説明になっていないか
- 変更容易性：この差分の構造で、次に想定される変更が局所化できそうか

## 除外・スコープ外
- linter / formatter / ast-grep などで自動検出できる機械的な指摘（命名規約、import 順、未使用変数 等）
- 使用者体験・API の使いやすさ（`usability` カテゴリで扱う）
- セキュリティ上の設計欠陥（`security` カテゴリで扱う）
- パフォーマンス特性に起因する設計判断（`performance` カテゴリで扱う）

## A. API 設計（リソース指向）

エンドポイント群がリソース指向（RESTful / AIP 的）な設計原則に則っているかをレビューする。
すべての項目は「自動判定が難しい設計判断」であり、CI で落ちるものは扱わない。

### A-1. リソースのモデリング
- エンドポイントが「**リソース（名詞）** + **HTTP メソッド（動詞）**」で表現されているか。
  `/getUser` `/createOrder` `/users/doSomething` のような **RPC 的な命名** になっていないか。
- リソース粒度は適切か。1 リソースが複数の集約境界を跨いでいないか。
  逆に、意味的に不可分なものを無理に分解していないか。
- **サブリソース化**の判断が妥当か。親なしで存在し得ない子だけをサブリソースにする（例: `/orders/{id}/items`）。
  独立してクエリ・共有される概念はトップレベルに置く。
- ネスト深度が 2-3 段を超えていないか。超える場合、フラット化（トップレベルリソース + クエリでフィルタ）の選択肢が検討されているか。
- コレクション名の複数形・単数形ポリシーがプロジェクト内で **一貫している** か
  （プロジェクト規約がある場合はそれに従う）。
- 「アクション（カスタムメソッド）」が必要な場合、リソースの下に `:verb` や `/actions/<name>` 等の
  **プロジェクト規約で定めた形式** に従っているか。無秩序に動詞エンドポイントが生えていないか。

### A-2. HTTP メソッドセマンティクス
- **GET** に副作用（書き込み / 状態変更 / 課金）がないか。
- **PUT / PATCH / POST** の使い分け:
  - `PUT` = べき等な全置換
  - `PATCH` = 部分更新（JSON Patch / JSON Merge Patch のどちらか方針を揃える）
  - `POST` = 非べき等な作成、またはリソース操作に収まらない RPC
- **DELETE** のべき等性。既に削除済みリソースへの DELETE で 404 を返すか 204 を返すかの方針が明確か。
- べき等性が期待される処理（決済・送信等）で `POST` を使っている場合、`Idempotency-Key` 等でリトライ安全性を担保しているか（A-7 と連動）。

### A-3. ステータスコード
- 2xx / 4xx / 5xx の帯を取り違えていないか。
  特に **サーバ側バグ由来の失敗を 4xx にして握りつぶしていないか**。
- 作成系で `201 Created` + `Location` を返すべきところを `200` で済ませていないか。
- **認可失敗と未存在の使い分け**（`403` vs `404`）が、情報漏洩方針と整合しているか
  （存在有無の漏洩を避けるため両方 404 にする等、プロジェクト方針を明文化してあるか）。
- `409 Conflict` / `412 Precondition Failed` / `422 Unprocessable Entity` / `429 Too Many Requests` など、
  意味的に適切な 4xx を選べているか（全部 `400` にまとめていないか）。

### A-4. URI / パラメータ設計
- **パスパラメータ = リソース識別子**、**クエリパラメータ = フィルタ / ソート / ページング** の役割分担が守られているか。
- ケース規約（kebab-case / snake_case / camelCase）が URI・クエリ・ボディで混在していないか。
- **バージョニング戦略**（URI 埋め込み `/v1/` / `Accept` ヘッダ / カスタムヘッダ）がプロジェクト方針と一致しているか。
  新旧の戦略が混在していないか。

### A-5. リクエスト / レスポンス形状
- レスポンスの **エンベロープ方針**（例: `{ data, meta }` か生の配列 / オブジェクトか）が
  API 全体で一貫しているか。今回の差分だけが例外になっていないか。
- **ページング方式**（オフセット / カーソル / キーセット）が、対象コレクションのサイズと
  並び順要件に対して妥当か。大規模コレクションに offset のみを許していないか。
- 一覧系エンドポイントに **デフォルト上限と max 上限** が設定されているか。
- `null` / フィールド欠損（undefined / キー自体の省略）/ 空配列 の扱いが一貫しているか。
  特に PATCH における「null で消したい」と「省略で変更しない」の区別が設計されているか。
- **エラーレスポンスが構造化** されているか（`application/problem+json` / 独自スキーマ）。
  エラーコード（machine-readable な識別子）が含まれているか。
- 機密情報（トークン・内部 ID・スタックトレース）がレスポンスに漏れていないか
  （詳細は `security` カテゴリで深掘りするが、API 設計レビュー時も気付けば指摘する）。

### A-6. コレクション操作
- **一括操作**の入力形式（body に配列 / `multipart` / 個別リクエストの並列呼び出し）が、
  サイズと一貫性要件に対して妥当か。
- 一括操作の **部分成功** をどう表現するか設計されているか
  （全件トランザクション vs 個別結果配列）。失敗件数・失敗理由が返せるか。
- フィルタ・検索エンドポイントの表現力と、GET の URL 長制限のトレードオフが考慮されているか
  （長大な検索条件を GET クエリに載せる vs `POST /resources:search` の選択）。

### A-7. 冪等性・楽観ロック
- リトライが想定される作成系・副作用系に `Idempotency-Key` 相当の機構があるか。
  キーの保持期間・衝突時挙動が定義されているか。
- 並行更新が起こり得るリソースに **楽観ロック**（`ETag` + `If-Match` / バージョンフィールド）が導入されているか。
  導入済みの場合、競合時の 412 応答形式がクライアントにとって復帰可能な情報を返しているか。

### A-8. 非同期処理 / 長時間処理
- 長時間処理（数秒超）を同期 API で返していないか。
  クライアントや LB のタイムアウトとの整合が取れているか。
- 非同期化する場合、`202 Accepted` + 進捗確認エンドポイント（`/operations/{id}` 等）または Webhook が用意されているか。
- 結果取得方式（ポーリング / Webhook / SSE / WebSocket）の選択が、利用文脈に対して妥当か。

### A-9. 後方互換性 / 進化
- 既存クライアントを壊す変更になっていないか:
  - フィールドの **削除・リネーム**
  - optional → required 化
  - enum 値の **追加**（受信側が closed 扱いの場合に破壊的）
  - ステータスコード・エラーコードの意味変更
- 破壊的変更となる場合、**deprecation の告知手段**（`Deprecation` ヘッダ / `Sunset` ヘッダ / ドキュメント）
  と期間が決まっているか。
- バージョン併走期間中、新旧の挙動差が整理されて実装されているか。

### A-10. コントラクトと実装の一致
- OpenAPI / GraphQL schema / gRPC proto 等 **コントラクト定義と実装が一致** しているか。
- コントラクトが **ソース・オブ・トゥルース** として扱われ、型・クライアント SDK が再生成で追従しているか
  （CLAUDE.md: コントラクト層を厳密に定義し実装層は再生成可能に保つ）。
- 手書きの型定義とコントラクトがドリフトしていないか。

## 除外・スコープ外
- インフラ構成そのもの（`infrastructure` カテゴリで扱う）
- UI 側の呼び出しパターン（`frontend` カテゴリで扱う）
- 認証メカニズムそのものの脆弱性・入力検証の網羅性（`security` カテゴリ）
- 計算量・N+1・スループットなどの性能特性（`performance` カテゴリ）
- 要求仕様の充足そのもの・エッジケース挙動（`functionality` カテゴリ）
- 失敗時の冪等性・整合性の全体像（`reliability` カテゴリ） ※本ファイルの A-7 は「API 設計として何を表出させるか」の観点に限定
